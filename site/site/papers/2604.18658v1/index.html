    Owner-Harm：AI Agent安全中被忽视的威胁模型¶

    Owner-Harm: A Missing Threat Model for AI Agent Safety

    元信息 内容
    作者 Dongcheng Zhang（蓝色光标传播集团）、Yiqing Jiang（同济大学）
    发表时间 2026年4月（预印本，审稿中）
    arXiv ID 2604.18658v1
    领域 AI Agent安全 · 威胁模型 · 提示注入 · 企业安全
    关键词 Owner-Harm, AI Agent Safety, Prompt Injection, Threat Model, Defense Gap

    摘要（Abstract）¶

    现有AI Agent安全基准过度聚焦于通用犯罪伤害（网络犯罪、骚扰、危险物质合成），对一种截然不同且商业后果严重的威胁类别存在系统性盲区：Agent伤害其自身的部署者（Owner-Harm）。

    真实事件佐证¶

    时间 事件 危害
    2024年8月 Slack AI被提示注入操纵，窃取部署组织私有频道token 凭证泄露
    2024年1月 Microsoft 365 Copilot被恶意日历邀请劫持，向外部转发敏感邮件 数据泄露
    2026年3月 Meta AI Agent未经授权发布内部论坛帖，暴露运营数据2小时 内部信息泄露

    共同特征：受害者是Agent的部署者，非第三方！

    核心贡献¶

    # 贡要 发现
    1 Owner-Harm威胁模型 首次正式定义Owner-Harm及其8个子类别
    2 防御缺口量化 同一安全系统：AgentHarm 100% TPR → Owner-Harm仅14.8% TPR（4/27; 95%CI: 5.9%-32.5%）
    3 缺口归因 零样本LLM分类器显示此差距非固有（62.7% vs 59.3%, gap=3.4pp），而是源于环境绑定的符号规则无法泛化到不同工具词汇表
    4 层级互补证据 Gate alone: 75.3% TPR → 加确定性后审计验证后: 85.3% TPR；Hijacking检测从43.3%→93.3%

    1 引言（Introduction）¶

    1.1 核心论点¶

    自主AI Agent投入生产系统的速度快于治理它们所需的安全框架。当前研究聚焦于防止Agent促进通用犯罪伤害——生成恶意软件、合成危险物质、生成骚扰内容。

    然而，实践中出现了一种定性不同且商业紧迫的威胁类别——Agent伤害部署和信任它们的组织本身。

    1.3 防御缺口的量化证据¶

    同一组合安全系统的表现对比： | 基准 | TPR（真阳性率） | FPR（假阳性率） | |------|----------------|----------------| | AgentHarm（通用刑事伤害，176个有害场景） | 100% ✅ | 0% ✅ | | AgentDojo注入任务（Owner-Harm，27个场景） | 14.8% (4/27) ❌ | — |

    → 成功拦截通用犯罪内容的约束条件（系统调用模式、毒性信号），对应用层Owner-Harm的覆盖率接近零——例如通过合法银行API发起的金融交易，或通过授权邮件工具转发的PII。

    2 Owner-Harm 威胁模型（Threat Model）¶

    2.1 形式化定义¶

    AI Agent建模: \(A = \langle M, \mathcal{T}, E, \pi \rangle\) - \(M\): 底层语言模型 - \(\mathcal{T}\): 可用工具集 - \(E\): 环境状态空间 - \(\pi\): Agent策略

    定义1：Owner（所有者）¶

    Owner \(O\) 是部署 \(A\)、授予资源访问 \(R_O\) 并对Agent行动承担问责的实体。关联： - 资源集 \(R_O\) - 环境状态的效用函数 \(U_O\) - 划定授权交互的信任边界 \(B_O\)

    定义2：Owner-Harm¶

    动作序列 \(\vec{a}\) 相对于Owner \(O\) 造成Owner-Harm 当且仅当： $\(OH(\vec{a}, O) := U_O(s') < U_O(s_0) \land \exists a_i \in \vec{a} : cause(a_i, s_0 \rightarrow s')\)$

    即：Owner效用降低 + Agent行动是该状态转换的近因或远因。

    定义3：直接Owner-Harm (DOH)¶

    $\(DOH(\vec{a}, O) := OH(\vec{a}, O) \land ProxCause(A, \vec{a})\)$ Agent是有害状态转换的近因（直接执行者）。

    定义4：间接Owner-Harm (IOH)¶

    $\(IOH(\vec{a}, O, V) := OH(\vec{a}, O) \land Manipulate(V, \pi, \vec{a})\)$ 对抗性Actor \(V\) 操纵Agent策略产生有害动作序列——即提示注入攻击模型。

    2.2 八大类 Owner-Harm 威胁分类¶

    编号 类别 说明 真实案例
    C1 凭证泄露 (Credential Leak) Agent泄露API密钥、密码、token Slack AI token外泄
    C2 基础设施暴露 (Infrastructure Exposure) 暴露服务器配置、网络拓扑、内部IP Copilot邮件转发暴露邮件基础设施
    C3 隐私暴露 (Privacy Exposure) 泄露用户PII或业务敏感数据 Meta Agent论坛帖子暴露运营数据
    C4 内圈泄露 (Inner Circle Leak) 向错误的人/系统披露受限信息 —
    C5 资产破坏 (Asset Destruction) 删除/修改关键文件或数据记录 —
    C6 工具外泄 via Tools (Exfiltration via Tools) 通过合法工具通道外传数据 日历注入→邮件外发
    C7 劫持 (Hijacking) Agent被外部控制执行非授权操作 Copilot日历劫持
    C8 未授权自治 (Unauthorized Autonomy) Agent超出预设权限范围自行决策 Meta Agent自发发帖

    3 实验与分析（Experiments and Analysis）¶

    3.1 缺口归因实验¶

    假设检验：Owner-Harm低检测率是因为该类别本质上更难，还是因为防御方法不适配？

    对照实验：零样本通用LLM分类器 - 在Owner-Harm基准上的表现：62.7% - 在AgentHarm基准上的表现：59.3% - Gap仅3.4pp → 差异不显著！

    结论：Gap并非源于Owner-Harm类别固有的难度，而是源于环境绑定的符号规则在不同工具词汇表间**泛化失败。

    3.2 层级互补性实验（300个Owner-Harm场景诊断基准）¶

    配置 TPR FPR Hijacking检测率
    仅Gate（语义门控） 75.3% 3.3% 43.3%
    Gate + 确定性后审计 85.3% (+10pp) — 93.3% (+50pp) ✅

    → 确定性后审计验证与语义门控推理正交（针对Hijacking/C7），通过互补检测维度大幅提升检测能力。

    3.3 SSDG概念框架（Symbolic-Semantic Defense Generalization）¶

    提出SSDG框架来组织观察：

    \[\text{Detection Rate} = f(\text{Information Coverage}, \text{Goal-Context Awareness}, \text{Action Alignment})\]

    其中Information Coverage（信息覆盖度）= 防护上下文集合满足攻击证据需求的分数

    两个受控SSDG实验： 1. 上下文剥夺实验：导致检测差距放大3.4倍（gap ratio R=3.60 vs R=1.06 with full context） 2. 上下文注入实验：揭示目标-上下文感知必要但不充分——结构化的目标-动作对齐（而非简单文本拼接）是有效Owner-Harm检测所必需的

    4 结论（Conclusion）¶

    4.1 核心结论¶

    1. Owner-Harm是被系统性忽视的威胁类别——现有安全基准几乎不覆盖
    2. 防御缺口显著（100% vs 14.8%）但非不可弥合——零样本LLM仅差3.4pp
    3. 需要上下文感知的安全机制——显式建模源所有权、信任边界和授权范围
    4. 层级防御互补性强——语义门控+确定性后审计可大幅提升检测率

    平台厂商 在Agent框架内置Owner-Harm防护层（类似OWASP for Agents）
