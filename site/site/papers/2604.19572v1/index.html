    TACO：基于观察上下文压缩的自演化高效终端Agent框架¶

    A Self-Evolving Framework for Efficient Terminal Agents via Observational Context Compression

    元信息 内容
    作者 Jincheng Ren, Siwei Wu†, Yizhi Li*, Kang Zhu, Shu Xu, Boyu Feng, Ruibin Yuan, Wei Zhang, Riza Batista-Navarro, Jian Yang‡, Chenghua Lin‡（曼彻斯特大学, MAP, HKUST, 北航）
    发表时间 2026年4月
    arXiv ID 2604.19572v1
    领域 终端Agent · 上下文压缩 · 自演化 · 软件工程
    关键词 Terminal Agent, Context Compression, Self-Evolving, Observational Compression, Plug-and-Play

    摘要（Abstract）¶

    随着模型能力的推进，研究越来越多地转向长程、多轮终端中心的Agentic任务——原始环境反馈常保留在交互历史中以支持未来决策。然而，反复保留此类反馈引入了大量冗余，并使累积token成本随步数二次方增长，阻碍长程推理。

    虽然观察压缩可以缓解此问题，但终端环境的异构性使得基于启发式或固定prompt的方法难以泛化。

    我们的方案：TACO¶

    提出 TACO（Terminal Agent COmpression）——一个即插即用、自演化的终端Agent压缩框架，从交互轨迹中自动发现并精炼压缩规则。

    主要结果¶

    基准 效果
    TerminalBench (TB 1.0 & 2.0) 一致提升1%-4%；相同token预算下再提升2%-3%
    Token开销 使用MiniMax-2.5时减少约10%
    其他基准 (SWE-Bench Lite, CompileBench, DevEval, CRUST-Bench) 一致的正向效果，跨Agent/模型/基准强泛化

    核心优势¶

    • 训练免费（training-free）：无需微调
    • 自演化：持续精炼压缩规则
    • 即插即用：可集成到任意现有Agent框架
    • 全局规则池：支持跨任务知识复用

    1 引言（Introduction）¶

    1.1 终端Agent的核心瓶颈¶

    主流终端Agent的一个关键瓶颈是：将原始终端输出直接送入后续决策。

    在真实的软件工程工作流中，这些观察通常包含大量冗余信息： - 冗长的日志 - 重复的build trace - 无关的命令输出

    ┌─────────────────────────────────────────────────┐
│  终端Agent每步的典型输入:                        │
│                                                 │
│  ████████████ 噪音（verbose logs, errors...）    │
│  ████████████ 冗余（重复的build output）         │
│  ████ 有效信息（task-relevant signals）          │
│                                                 │
│  → 不仅增加计算成本，还掩盖关键信号               │
│  → 最终损害长程推理和任务性能                    │
└─────────────────────────────────────────────────┘

    1.2 现有方法的局限¶

    方法类型 代表 局限
    预定义静态压缩 Seed Rules (6条启发式) 固定规则无法适应环境变化
    专家手工规则 High-Quality Rules (200条) 人力成本高，难维护
    LLM-based压缩 通用LLM摘要 不稳定，跨任务波动大
    训练方法 SWE-Pruner 需额外训练，局限于SWE风格任务

    根本原因：终端环境高度异构——所需的压缩模式在不同仓库、命令和执行状态下差异巨大。

    1.3 TACO的设计哲学¶

    有效的终端输出压缩应适应当前环境并在Agent交互期间持续精炼，而非依赖固定启发式或任务特定的压缩器训练。

    2 方法：TACO框架（Methodology）¶

    2.1 整体架构¶

    ┌─────────────────────────────────────────────────────┐
│                    TACO 框架                          │
│                                                     │
│  ┌─────────────┐    ┌─────────────────────────────┐│
│  │ Global Rule  │◄───│   Self-Evolution Loop       ││
│  │ Pool (GRP)   │    │  (跨任务共享的可复用规则)      ││
│  │             │    └─────────────┬───────────────┘│
│  │ • Rule α₁   │                  │                │
│  │ • Rule α₂   │    ┌─────────────▼───────────────┐│
│  │ • Rule α₃   │    │  Task-Specific Adapter      ││
│  │ • ...        │    │  (从GRP初始化 + 在线适配)    ││
│  └──────┬──────┘    └─────────────┬───────────────┘│
│         │                           │                │
│         ▼                           ▼                │
│  ┌───────────────────────────────────────────┐    │
│  │      Terminal Observation Compressor      │    │
│  │                                         │    │
│  │   Raw Observation Oₜ                      │    │
│  │        │                                  │    │
│  │        ▼                                  │    │
│  │   [Apply Task-Specific Rules]             │    │
│  │        │                                  │    │
│  │        ▼                                  │    │
│  │   Compressed Observation Ôₜ               │    │
│  │   (保留任务相关信息，去除冗余噪声)           │    │
│  └───────────────────┬───────────────────────┘    │
│                      │                            │
└──────────────────────┼────────────────────────────┘
                       │
                       ▼
              送入 Agent 下一步决策

    2.2 自演化机制（Self-Evolution Mechanism）¶

    三步循环：

    Step 1: DISCOVER（发现）
  从终端交互轨迹中自动发现压缩模式
  - 识别重复出现的冗余模式
  - 提取可泛化的压缩规则

Step 2: REFINE（精炼）
  在Agent交互过程中持续改进规则
  - 监测压缩效果（性能 vs token节省）
  - 根据反馈调整规则参数

Step 3: REUSE（复用）
  通过全局规则池实现跨任务知识迁移
  - 新任务初始化时从GRP继承相关规则
  - 成功的规则回馈更新GRP

    2.3 收敛判据¶

    TACO引入了一个实用的自演化收敛判据： - 何时停止演化？ 当规则演化过程趋于稳定时 - 避免无限增长规则池或过拟合当前任务

    3 实验结果（Experiments）¶

    3.1 TerminalBench 结果¶

    集成TACO到现有Agent框架的效果：

    主干模型 TerminalBench 1.0 TerminalBench 2.0
    DeepSeek-V3.2 +1~4% +1~4%
    MiniMax-M2.5 +1~4% 且 -10% token ✅ +1~4% 且 -10% token ✅
    Qwen3-Coder-480B +1~4% +1~4%
    Qwen3系列其他 +1~4% +1~4%

    相同Token预算下的额外收益：+2%~3% 准确率提升 → 说明去除冗余终端上下文同时提升了效率和效果

    3.2 跨基准泛化¶

    基准 类型 TACO效果
    SWE-Bench Lite 软件工程 正向 ✅
    CompileBench 编译任务 正向 ✅
    DevEval 开发评估 正向 ✅
    CRUST-Bench CRUST命令 正向 ✅

    跨不同Agent框架和主干模型，学习的压缩规则都表现出良好的迁移能力。

    3.3 Token效率¶

    使用MiniMax-2.5时： - Token开销减少约10% - 性能反而提升 → 证明噪声信息的去除有助于模型更好地聚焦关键信号

    4 讨论（Discussion）¶

    4.1 TACO为何有效？¶

    因素 解释
    自适应 规则适应当前仓库/命令/状态的特殊模式
    在线学习 在交互过程中持续改进，不是一次性固定方案
    跨任务迁移 全局规则池积累了通用的压缩智慧
    无训练成本 即插即用，无需微调任何模型

    4.2 与相关工作的关系¶

    方面 TACO的位置
    vs SWE-Pruner 训练free，适用范围广（不限SWE任务）
    vs 上下文管理工具 自动发现规则而非手动设计
    vs 通用LLM摘要 结构化规则驱动，更稳定可控
    vs Self-Evolving Agents (MetaClaw等) 将自演化思想首次应用于上下文压缩场景

    4.3 局限性与未来工作¶

    1. 规则质量依赖初始轨迹：如果Agent本身就表现差，发现的压缩规则也可能质量不高
    2. 收敛判据的经验性：当前收敛判据基于启发式
    3. 尚未考虑多模态观察：未来可扩展到GUI终端等场景
    4. 与其他优化目标的权衡：压缩可能与探索性行为存在张力

    5 结论（Conclusion）¶

    TACO证明了： 1. ✅ 自演化的、任务感知的压缩对终端Agent有效 2. ✅ 训练-free即可获得一致的性能和效率提升 3. ✅ 强跨Agent/模型/基准泛化能力 4. ✅ 为缓解终端环境中的上下文饱和瓶颈提供了通用可扩展的解决方案

    最终启示：终端Agent不需要更多的上下文窗口——它们需要更好的上下文。
