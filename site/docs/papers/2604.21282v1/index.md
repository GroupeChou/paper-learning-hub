# 策略性异构多智能体架构：面向成本高效的代码漏洞检测

## Strategic Heterogeneous Multi-Agent Architecture for Cost-Effective Code Vulnerability Detection

Zhaohui Geoffrey Wang

南加州大学

AAMAS 2026

---

## 摘要

自动代码漏洞检测对软件安全至关重要，但现有方法面临检测精度与计算成本之间的根本权衡。我们提出一种受**博弈论**启发的异构多智能体架构，结合云端 LLM 专家和本地轻量级验证器。

**"3+1"架构：**
- **3 个云端专家智能体**（DeepSeek-V3）：从互补视角分析代码——代码结构、安全模式、调试逻辑——**并行**执行
- **1 个本地验证器**（Qwen3-8B）：以**零边际成本**进行对抗性验证

**双层博弈框架：**
1. **合作博弈（Layer 1）**：三个专家形成具有超加性价值的联盟
2. **对抗验证博弈（Layer 2）**：验证器作为独立对手，通过捕捉误报提高精度

**实验结果（NIST Juliet 测试套件，262 个样本，14 个 CWE 类型，漏洞/良性平衡）：**
- **77.2% F1**，62.9% 精度，**100% 召回率**，**$0.002/样本**
- 优于单专家 LLM 基线（F1 71.4%）和 Cppcheck 静态分析（MCC 0）
- 对抗性验证器显著改善精度 **+10.3 个百分点**（p < 10⁻⁶）
- 并行执行实现 **3.0× 加速**

## 1 引言

代码漏洞检测是软件安全的关键挑战。传统静态分析工具（Cppcheck）速度快、成本低但语义理解有限；单智能体 LLM 精度更高但召回率不足且成本高昂。

**核心问题**：如何在多个智能体之间分配异构计算资源，以实现检测质量与运营成本之间的最优权衡？

**关键洞见**：漏洞检测受益于两种不同的能力——(1) 需要强大模型的深度语义分析，和(2) 可由较浅模型执行的一致性检查。不对称性天然适合异构架构。

---

## 2 相关工作

**静态分析**：Cppcheck、Coverity——高误报率，缺乏语义理解。

**基于 LLM 的漏洞检测**：单智能体方法（GPT-4）检测率高但成本高。PrimeVul 基准揭示：SOTA 模型在精心去重的真实数据集上仅达到 ~3-35% F1。

**多智能体方法**：VulTrial（ICSE 2026）、MulVul、MultiVer。但与我们的工作不同，它们使用同质模型池（所有智能体共享相同昂贵的 API），我们结合付费云端专家与免费本地验证器，使成本成为**头等设计目标**。

**博弈论与 LLM 多智能体系统**：GTBench、ECON 框架等新兴方向。我们的使用方式：博弈论作为**设计框架**（分析工具）而非假设 LLM Agent 的策略行为。

## 3 方法论

### 3.1 问题形式化

**定义 1（多智能体漏洞检测博弈）**：系统 M = ⟨A, S, C, Q, π⟩，其中 A 是异构智能体集合，S 是策略空间，C 是成本函数，Q 是质量函数，π 是支付函数。

机制设计者的问题：选择智能体类型、角色和交互协议，以在低成本下诱导高 Q 的期望均衡。

### 3.2 双层博弈结构

**Layer 1：专家合作博弈（联盟博弈）**

**定义 2（专家联盟博弈）**：G₁ = ⟨N, v⟩，其中 N 是专家智能体集合，v 是特征函数。

联盟价值：v(S) = w₁·Q(S) − w₂·C(S)

**命题 1（多样性联盟的超可加性）**：令 S,T ⊆ N 为具有互补分析视角的不交联盟。若 S 中的智能体检测到漏洞集 V_S，T 检测到 V_T，则：
Q(S∪T) ≥ max(Q(S), Q(T))
当 V_S ⊈ V_T 且 V_T ⊈ V_S 时严格不等式成立。

**Shapley 值分析**：量化每个专家对联盟的边际贡献。

**Layer 2：对抗验证博弈（扩展形式博弈）**

**定义 3（验证博弈）**：G₂ = ⟨P, S_E, S_V, π_E, π_V⟩

**定理 1（验证均衡）**：
(i) 若验证器独立检测误报概率 p_fp > 0 且不引入新误报，则当 FP > 0 时 P_system > P_experts
(ii) 当拒绝惩罚超过高质量输出的成本差异时，(s*_E=high, s*_V=accept when consistent) 形成纳什均衡

### 3.3 "3+1"架构

**专家智能体（DeepSeek-V3）：**
- **代码分析师**：数据流、控制流、漏洞入口点、内存操作
- **安全专家**：CWE 分类法匹配、已知漏洞模式、严重性评估
- **调试专家**：错误处理、边界条件、边缘情况、未定义行为

每个专家输出结构化报告：VULNERABILITY_FOUND, CWE_IDs, SEVERITY, EVIDENCE, CONFIDENCE。

**验证器（Qwen3-8B）：**
- 零边际成本（本地 GPU）
- 不同模型家族（Qwen3 vs DeepSeek），减少相关错误
- 全信息：接收原始代码和所有三个专家报告

### 算法 1：3+1 多智能体检测

```
输入：代码 x, 专家 E={e1,e2,e3}, 验证器 V
输出：漏洞预测，CWE 分类

// Layer 1: 合作博弈
{r1, r2, r3} ← ParallelAnalyze(E, x)   // 联盟 v(N)
CWE_pool ← ∪ ExtracCWEs(r_i)           // 超可加并集

// Layer 2: 验证博弈
v ← Verify(V, x, r1, r2, r3)           // 审计机制
if v 提供 FINAL_VULNERABILITY then
    return v.vulnerability, v.CWE_IDs   // 验证器覆盖
else
    return MajorityVote(r1,r2,r3), CWE_pool
```

## 4 实验

### 4.1 数据集

NIST Juliet Test Suite v1.3，262 个函数级样本：
- 平衡类别：132 个漏洞 + 130 个良性
- 14 个 CWE 类型（内存安全、整数错误、空指针、注入、资源管理等）
- 平均每样本 38.5 行代码

### 4.2 主要结果

**表 1：主要结果（262 个样本）**

| 配置 | 精度 | 召回率 | F1 | FPR | MCC |
|------|------|--------|-----|-----|-----|
| **3+1 并行+验证器** | **.629** | **1.00** | **.772** | .600 | .501 |
| 3+1 并行-验证器 | .526 | 1.00 | .689 | .915 | .211 |
| 单专家 | .555 | 1.00 | .714 | .815 | .320 |
| Cppcheck | .504 | 1.00 | .670 | 1.00 | .000 |

**表 2：成本与延迟**

| 配置 | 成本/样本 | 时间/样本 |
|------|----------|----------|
| 3+1 并行+验证器 | $0.0021 | 68.1s |
| 单专家 | $0.0006 | 14.1s |
| Cppcheck | $0 | <0.1s |

### 4.3 RQ1：专家联盟的超可加性

**表 3：对抗性验证器效果**

| 指标 | +验证器 | -验证器 | Δ | p 值 |
|------|--------|--------|---|------|
| 精度 | 62.9% | 52.6% | **+10.3%** | <10⁻⁶ |
| 召回率 | 100% | 100% | 0% | — |
| F1 | 77.2% | 68.9% | +8.3% | — |
| FPR | 60.0% | 91.5% | **-31.5%** | — |

超可加性体现在**CWE 识别精度**而非召回率：三个专家视角的并集实现 **100% CWE 匹配率**（每个检测到的漏洞都被正确分类），远高于单个专家的匹配率。

### 4.4 RQ2：对抗性验证器影响（定理 1 验证）

验证器**从 119 个误报中过滤掉 41 个**（降至 78），零样本被错误地从正确变为不正确。MCC 从 0.211 跃升至 **0.501**，证实验证器显著提升了判别力。

**博弈论解释**：这验证了 Layer 2 的检查博弈结构。不同模型家族（Qwen3 vs DeepSeek）的验证器作为独立审计者，捕捉同模型验证会遗漏的不一致性。

### 4.5 RQ3：并行执行效率

| 指标 | 并行 | 串行 | 增益 |
|------|------|------|------|
| 专家时间/样本 | ~15s | ~45s | **3.0×** |
| 总时间/样本 | 68.1s | 92.2s | 1.4× |
| F1 | 77.2% | 78.3% | ≈相等 |

### 4.6 每 CWE 分析

**表 5：每 CWE 性能**

| CWE | 类别 | FPR | 说明 |
|-----|------|-----|------|
| CWE-476 | 空指针解引用 | **11%** | 补丁添加显式检查，结构区分明显 |
| CWE-252 | 未检查返回值 | 20% | 同样是显式检查模式 |
| CWE-400 | 资源耗尽 | **100%** | 语义漏洞，补丁保留相似结构 |
| CWE-416 | 释放后使用 | 100% | LLM 难以区分的细微控制流变化 |

## 5 博弈论作为分析工具

关键主张：博弈论框架对多智能体 LLM 系统行为提供**正确且有用的预测**。

| 预测 | 来源 | 实证验证 |
|------|------|---------|
| 多样性联盟的超可加价值 | 合作博弈 | 100% CWE 匹配率 ✅ |
| 独立验证提高精度 | 对抗博弈 | p < 10⁻⁶ ✅ |
| 异构云-本地分配主导同质设计 | 机制设计 | 零成本精度增益 ✅ |

**Shapley 值验证器**：ϕ_V = v({1,2,3,V}) − v({1,2,3}) = +0.106（F1 项）。验证器的边际成本为零 → ϕ_V/C_V → ∞——添加自由本地验证器始终是理性选择。

## 6 结论

博弈论启发的异构多智能体漏洞检测架构：

| 指标 | 数值 |
|------|------|
| F1 | **77.2%** |
| 精度提升（验证器） | **+10.3 pp** |
| 召回率 | **100%** |
| 成本 | **$0.002/样本**（500 个函数仅需 $1） |
| 加速 | **3.0×**（并行 vs 串行） |

**局限性**：基于合成 Juliet 测试套件；真实漏洞更复杂；误报率仍较高。未来工作包括：真实数据集评估、迭代专家-验证器对话、动态 Agent 选择、CI/CD 集成。
