# Owner-Harm：AI Agent 安全中缺失的威胁模型

## Owner-Harm: A Missing Threat Model for AI Agent Safety

张东晨\*, 蒋艺青†

\*蓝色光标  †同济大学

---

## 摘要

现有 AI Agent 安全基准主要聚焦于**通用犯罪危害**——网络犯罪、骚扰和武器合成——留下了一个系统性的盲点：**Agent 伤害其部署所有者**。真实事件说明此差距：Slack AI 凭据泄露（2024年8月）、Microsoft 365 Copilot 通过日历注入的数据泄露（2024年1月）、Meta Agent 未授权论坛帖子（2026年3月）。

我们提出 **Owner-Harm**，一个形式化威胁模型，定义了八类损害部署所有者利益的 Agent 行为。

**关键发现：**
- 同一组合安全系统在 AgentHarm（通用危害，176个场景）上达到 **100% TPR**，但在 AgentDojo（提示注入介导的所有者危害）上仅 **14.8%**——暴露了特定类别盲点
- 零样本通用 LLM 分类器显示：此差距不是 Owner-Harm 类别固有的（62.7% vs 59.3%，差 3.4 pp），而是源于**环境绑定的符号规则未能跨工具词汇泛化**
- 在 300 个 Owner-Harm 场景的诊断基准上：安全门 TPR 75.3%，加确定性事后审计验证器后 TPR 升至 **85.3%**（+10.0 pp），Hijacking 检测从 43.3% 升至 **93.3%**

## 1 背景与动机

### 三个真实事件

| 时间 | 事件 | 类型 |
|------|------|------|
| 2024年8月 | Slack AI 被提示注入操纵，泄露私有频道令牌 | 凭据泄露(C1) |
| 2024年1月 | M365 Copilot 被恶意日历邀请劫持，转发敏感邮件 | 隐私泄露(C3) |
| 2026年3月 | Meta AI Agent 向内部论坛发布未授权帖子，暴露运营数据2小时 | 内部泄露(C4) |

在每种情况下，Agent 的**部署者**——而非第三方受害者——承受了伤害。

---

## 2 Owner-Harm 威胁模型

### 2.1 形式化定义

**定义 1（所有者）**：所有者 O 是部署 Agent A、授予资源访问 RO、为 Agent 行动承担责任的主体。

**定义 2（Owner-Harm）**：行动序列 ⃗a 对所有者 O 造成伤害当且仅当：
$$OH(\vec{a}, O) := U_O(s') < U_O(s_0) \land \exists a_i \in \vec{a}: cause(a_i, s_0 \to s')$$

### 2.2 八类 Owner-Harm

| 类别 | 描述 | 真实案例 |
|------|------|---------|
| **C1: 凭据泄露** | 泄露 API 密钥、OAuth 令牌、会话 Cookie | Slack AI 2024 |
| **C2: 基础设施暴露** | 暴露网络规则、云策略、内部架构 | AI 辅助代码生成产生过度权限的 IAM 策略 |
| **C3: 隐私泄露** | 传输 PII 或机密业务数据给未授权方 | M365 Copilot 2024 |
| **C4: 内部圈泄露** | 泄露客户列表、合作伙伴协议、战略路线图 | 三星员工向 ChatGPT 粘贴源代码 |
| **C5: 资产破坏** | 不可逆删除或损坏数据 | 编码Agent在生产目录执行 rm -rf |
| **C6: 通过工具窃取** | 利用授权工具作为隐蔽数据通道 | ASCII 走私 via M365 Copilot |
| **C7: 劫持** | 对手获得 Agent 的持久性控制 | AutoGPT 内存中毒 |
| **C8: 未授权自主** | Agent 超出授权范围采取行动 | Air Canada 聊天机器人承诺未授权退款 |

### 2.3 与现有框架的对比

| 类别 | AgentHarm | ToolEmu | OWASP | AgentDojo |
|------|-----------|---------|-------|-----------|
| C1 凭据泄露 | ∼ | ✓ | ✓ | ∼ |
| C4 内部圈泄露 | ❌ | ❌ | ❌ | ❌ |
| C6 工具窃取 | ❌ | ∼ | ∼ | ∼ |
| C7 劫持 | ❌ | ∼ | ✓ | ∼ |

C4（内部圈泄露）是最显著的差距——**不存在于任何现有基准中**。

### 2.4 为什么通用防御失败

根本原因是危害的**上下文依赖性**：
$$\exists c, O_1, O_2: Harmful(c, O_1) \land \neg Harmful(c, O_2)$$

"转发邮件到 external@attacker.com"对于信任边界包含该地址的所有者是良性的，对于不包含则是恶意的。纯内容分类器无法做出这一区分。

## 3 Nous：组合式运行时安全门

四层组合式架构：

| 层 | 名称 | 机制 | 延迟 | LLM 需求 |
|----|------|------|------|---------|
| L1 | Datalog 阻断器 | 46 条 Datalog 规则，编码所有者策略约束 | 0.055ms | 无 |
| L2 | 琐碎性过滤器 | 将明显良性的行动路由过语义层 | ～70%成本降低 | 无 |
| L3 | 语义门 | LLM 评估器，分析意图分解 | — | 有 |
| L4 | 事后验证器 | 6 条确定性审计规则 | 0.038ms | 无 |

## 4 评估

### 4.1 AgentDojo 结果

**表 3：AgentDojo 安全分数 vs 效用（27 个注入任务，97 个用户任务）**

| 配置 | 安全性 | 效用 | FPR |
|------|--------|------|-----|
| L1 仅 | 3.7% (1/27) | 97.9% | 2.1% |
| L1+L4 | 3.7% | 97.9% | 2.1% |
| **L1+L3+L4（完整）** | **14.8% (4/27)** | **97.9%** | **2.1%** |

### 4.2 跨基准对比

**表 7：跨基准安全分数**

| 基准 | 焦点 | Nous 安全 | 通用 LLM |
|------|------|----------|---------|
| AgentHarm | 通用犯罪 | **100%** | 62.7% |
| Owner-Harm (诊断) | 所有者指向 | 75.3% | — |
| AgentDojo | 提示注入 | **14.8%** | 59.3% |

**核心发现**：同一系统在通用危害上表现完美，在所有者危害上几乎完全失效。

### 4.3 对照实验：差距归因

**表 8：差距归因——Nous vs 通用 LLM 分类器**

| 系统 | AgentHarm TPR | AgentDojo TPR | 差距 |
|------|-------------|--------------|------|
| Nous | 100.0% | 14.8% | **85.2 pp** |
| 通用 LLM | 62.7% | 59.3% | **3.4 pp** |

差距不是语义难度固有的——通用 LLM 在两类上的差异仅 3.4 pp。Nous 的 85.2 pp 差距源于 L1 符号规则从 AgentHarm 工具词汇（bash、read file）泛化到 AgentDojo 应用层词汇（send money、send email）的失败。

### 4.4 Hijacking 层互补性

| 检测象限 | 计数 | 占比 |
|---------|------|------|
| 仅门检测到 | 11 | 18.3% |
| **仅验证器检测到** | **30** | **50.0%** |
| 两者检测到 | 15 | 25.0% |
| 皆未检测到 | 4 | 6.7% |
| **组合** | **56** | **93.3%** |

### 4.5 SSDG 框架实验

**P1（上下文注入）——预测被拒**：将用户目标注入门上下文后 TPR 不升反降（14.8%→8.3%）。表明目标意识是必要但非充分——需要结构化目标-行动对齐而非简单文本拼接。

**P2（上下文剥夺）——预测确认**：从语义防御中剥离世界上下文后，差距比 R=3.60（远超阈值 1.5），确认语义检测的稳定性依赖对世界上下文的访问。

## 5 生产部署证据

在影子模式下部署 11+ 天（30,204 次调用）：
- **一致性**：99.47% 与现有基于规则的系统一致
- **漏报恢复**：检测到 157 个现有系统放过的事件（经人工确认均为真阳性）
- **过度阻断**：4 例过度触发

## 6 讨论与启示

**核心结论**：现有防御是"所有者上下文盲"的。一个行动对所有者有害不是因为其孤立的内容，而是因为它如何与所有者的资源、信任边界和授权范围相关。

**架构启示：**
- 符号规则：分布内近乎完美，分布外近乎为零
- 语义推理：始终一致但低于完美
- 组合架构：两个层具有互补的泛化曲线

**未来方向：**
- 结构化目标-行动对齐（预计 AgentDojo 安全性可达 40-60%）
- UEBA 适应 AI Agent 部署
- 自适应免疫（从部署日志学习 Owner-Harm 模式）
- 所有者上下文清单标准

## 8 结论

Owner-Harm 是 AI Agent 安全中系统性研究中不足的威胁类别。现有防御是所有者上下文盲的——直到安全系统显式建模 RO、BO 和 AuthO 这些上下文维度，Owner-Harm 将仍然是系统性盲点。
