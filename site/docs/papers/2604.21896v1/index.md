## 摘要

本文提出了一种全新的 **AI 游戏编程范式（paradigm）**，利用大语言模型（LLMs）来扩展和操作化 Claude Shannon 的游戏博弈机器分类法。该范式的核心是 **Nemobot**——一个交互式智能体工程环境（interactive agentic engineering environment），它允许用户创建、定制和部署由 LLM 驱动的游戏智能体（game agents），同时主动与 AI 驱动的策略进行交互。

集成在 Nemobot 中的基于 LLM 的聊天机器人（chatbot），展示了其在 **四类不同游戏** 上的能力：

| 游戏类别 | LLM 的角色与能力 |
|----------|------------------|
| **字典型游戏** | 将状态-动作映射压缩为高效、泛化的模型以实现快速适应 |
| **严格可解游戏** | 运用数学推理计算最优策略，并生成人类可读的决策解释 |
| **启发式游戏** | 综合经典极小极大算法（minimax）的见解与众包数据来合成策略 |
| **学习型游戏** | 利用带人类反馈的强化学习（RLHF）和自我批评（self-critique），通过试错学习和模仿学习迭代优化策略 |

Nemobot 通过提供一个**可编程环境**进一步放大了这一框架——用户可以在其中实验工具增强的生成（tool-augmented generation）和战略性游戏智能体的微调（fine-tuning）。从战略游戏到角色扮演游戏，Nemobot 展示了 AI 智能体如何通过整合众包学习（crowdsourced learning）和人类创造力来实现一种形式的**自编程（self-programming）**，迭代地优化其自身逻辑。这代表了向**自编程 AI（self-programming AI）**长期目标迈出的一步。

[扩展] 本文的核心创新在于将 Shannon 1950 年代提出的经典游戏机器分类法与现代 LLM 能力相结合，构建了一个既具有教育价值又具备研究深度的编程框架。它不是简单地用 LLM 来玩游戏，而是让 LLM 成为"游戏编程"的一部分，从而实现了从"用 AI 玩游戏"到"用 AI 教人做游戏 AI"的范式跃迁。

## Section 1 — 引言（Introduction）

### 中文翻译

人工智能先驱 **Claude Shannon** 在其开创性著作 [2] 中提出了一个发人深省的问题：

> "我们能否对数字计算机进行编程，使得（最终）它遵循的指令中有 99% 是由计算机自己编写的，而不是当前程序中那百分之几？"

这个问题概括了**自编程系统（self-programming systems）**的愿景，这一概念至今仍是 AI 研究的核心挑战。

同样，Marvin Minsky 在 [3] 中论证道：**编程的结构化规则不是限制，而是用于提炼和阐明理解不清晰想法的工具**——类似于艺术家和作曲家如何利用形式约束来表达创造力。

Shannon 关于计算机自主生成自身指令的愿景在**智能体工程（agentic engineering）**中变得日益相关，如今大语言模型（LLMs）产生大量代码，驱动着自演化 AI 系统和游戏等交互式应用的发展。

![Figure 1: 游戏博弈 AI 中的众包与策略优化](assets/page-001-img-01.png)

*Figure 1: 游戏博弈 AI 中的众包与策略优化。LLM 为游戏博弈智能体生成优化策略，而来自人类玩家交互的游戏状态和结果被反馈回来训练 LLM，通过众包创建了一个自我强化的改进循环。[扩展] 这张图揭示了 Nemobot 的核心闭环架构：LLM 不是一次性生成策略就结束了，而是通过与人类玩家的持续交互不断获得反馈数据，形成类似"飞轮效应"的正向循环。*

最近的综述如 [4]、[5] 探索了利用 LLMs 提升战略 AI 游戏智能体适应性和推理能力的进展。针对特定应用的研究——如在《外交》（Diplomacy）游戏中结合语言模型与战略推理 [6]，以及使用 LLMs 作为测试员测量游戏难度 [7]——证明了 LLMs 在不完美信息游戏中的多样性。像 ChessGPT [8] 和 AlphaStar [9] 这样的方法架起了策略学习与语言建模之间的桥梁，强调了协同强化学习与自然语言处理的重要性。Suspicion-Agent [10] 等系统的开发进一步将心智理论感知推理（theory-of-mind-aware reasoning）与 OpenAI 的 GPT-4 相集成，扩展了不完美信息设置下多智能体交互的视野。

**游戏棋盘作为微观世界**：离散、可观的世界状态为研究 LLM 中世界模型的表征学习提供了结构化、可控的测试床。当在游戏对局记录和规则上训练时，LLM 可以学习隐式表征（implicit representations），不仅编码合法走法，还编码时间依赖性、隐藏状态和概率化的对手策略。例如，[11] 表明国际象棋对弈使 LLM 能够跟踪棋盘状态并预测走法，证明了战略环境可以诱导游戏动态的有意义的内部表征。这与 [12] 的发现一致——他们表明在合成序列预测任务上训练的 Transformer 会自发形成捕获潜在环境动力学的内部世界模型，而无需显式监督。

为了严格评估这些学习表征的保真度（fidelity），[13] 提出了系统性方法来探测生成模型是否对未观测变量保持一致的信念并遵守转移规则。综合而言，这些研究表明**游戏对弈可以驱动 LLM 中世界模型的表征学习**，尽管这些表征通常仍是近似的且缺乏显式落地（grounding）。

[扩展] 这段论述揭示了一个深刻的洞察：游戏之所以适合作为 LLM 世界模型研究的测试床，是因为它们提供了"干净的"环境——状态空间有限、规则明确、反馈即时。这与开放域的自然语言处理形成鲜明对比，后者充满歧义和噪声。作者暗示，如果 LLM 能在游戏中学会形成良好的内部表征，这种能力可能迁移到更复杂的现实世界任务中。

为了提炼和利用这些表征，**AI 游戏编程提供了一种强大的结构化世界模型学习范式**。通过设计游戏模拟器和编程接口，我们可以创建这样的环境：LLM 智能体必须学习显式的、可验证的表征才能成功。例如，可以设计一个智能体工程框架，通过结构化 API 与游戏引擎交互，强制 LLM 显式表示棋盘状态、动作空间和对手模型以生成有效的函数调用。这种方法使用**可编程的反馈循环**将抽象内部表征落地：LLM 的预测立即被游戏引擎验证或纠正，实现其世界模型的错误驱动精化（error-driven refinement）。通过与这些编程接口的反复交互，AI 智能体可以学习更一致、准确且可泛化的表征，有效地将对弈转化为一种**自监督表征学习**形式。

这些可编程方法产生的世界模型不仅更具可解释性（interpretable），而且对下游战略推理更具可操作性（actionable），从而缩小了涌现内部表征与外部可验证游戏语义之间的差距。这种下游战略推理可以通过改进语言模型推理的已有方法进一步增强：**自一致性（Self-consistency）**[14] 通过结构化推理链精化决策质量，而 **ReAct** [15] 和 **思维图（Graph of Thoughts）** [16] 等框架为复杂环境中的推理和规划提供了形式化机制。此外，检索增强的战略博弈（retrieval-augmented strategic play）[17] 展示了语言模型如何动态融合外部知识来引导探索并通过基于 LLM 的智能体改善游戏表现。

最近 **AI 辅助编程**的进展正在改变复杂游戏 AI 智能体的开发。像 OpenAI GPT 这样的基础模型利用先进的 NLP 和 LLM 能力来解释人类指令并生成功能代码 [18]。这使得 GitHub Copilot [19]、Xcode Copilot [20], [21]、OpenAI Codex [22] 和 Claude Code [23] 等工具能够支持程序员快速原型设计、调试和优化智能体逻辑，从而加速了游戏 AI 的推理循环设计、工具集成和战略模块开发。公开可用 LLM 系统（如 OpenAI GPT [24] 和 Google Gemini [25]）不断演进的**多模态能力**进一步扩展了这一潜力，加强了 AI 辅助编程、战略推理和交互式游戏之间的协同效应。像 CAMP [26], [21] 这样使用基于上下文的 RAG 方法的最新框架展示了本地-云架构（local-cloud architectures）和检索机制如何进一步增强 AI 辅助编程和软件测试的可靠性和上下文感知能力。

这些发展使我们能够通过**智能体工程的新编程范式**重新审视 Shannon 的问题——其中学习到的表征被程序化地利用（而非依赖完整世界模型）来创建混合的人机解决方案。

[扩展] 这里作者提出了一个关键区分："程序化地利用学习到的表征" vs "依赖完整世界模型"。前者承认 LLM 的表征是不完美的（approximate），但通过编程接口将其约束和落地；后者则要求系统拥有完美的世界模型——这在实践中几乎不可能。这种务实的折中是 Nemobot 设计哲学的核心。

一个自然的问题是：**如果现代 LLM 已经能在直接提示下玩井字棋或 Nim 这样的游戏，专用编程框架的价值何在？**

区别在于**智能（intelligence）与编程（programming）的差异**：
- 提示 LLM 玩游戏产生的是**非确定性的、往往不可复现的行为**，对用户来说是不透明的，难以系统地调试或改进
- Shannon 的分类法 [2], [27] 是在游戏博弈智能力主要基于确定性算法的时代开发的；然而当前的 LLM 系统本质上是**非确定性的**
- 缺乏的是一种**结构化编程范式**，它在可复现的、可教育的、可扩展的框架中利用 LLM 能力

**Nemobot 弥补了这一差距**，提供了一个可编程环境，其中 LLM 函数被视为**模块化的、可检查的组件**（类似于子例程/subroutine），用户可以设计、测试、精炼和共享。Nemobot 不仅仅是编排 LLM API 调用，而是通过**可编程提示工程（programmable prompt engineering）、众包策略精炼（crowdsourced strategy refinement）和分层课程体系（tiered curriculum）**操作化了 Shannon 的分类法——这个课程体系将学习从简单的字典型游戏脚手架式地支撑到高级强化学习智能体。

这将贡献定位为不是对 LLM 服务的包装，而是**一个研究框架**，调查经典游戏博弈范式如何能被系统地用 LLM 扩展以服务于 AI 研究和教育。IEEE Transactions on Games 中的最新工作进一步激励了这一方向：Phillips 等 [28] 使用 LLM 实现了目标导向的 NPC 交互；Bassanelli 等 [29] 强调了有原则的游戏化设计（principled gamification design）对有效学习成果的重要性。

本文**重新审视 Shannon 的游戏博弈机器分类法** [2], [27] 并使用现代 LLM 能力对其进行扩展，引入了战略游戏开发的编程范式。Shannon 的四类如图 2 所示：

![Figure 2: Shannon 的四种游戏博弈机器类型及其典型应用示例](assets/page-001-img-02.png)

*Figure 2: Shannon 的四种游戏博弈机器类型及其典型应用示例。[扩展] 这个分类法是整个 Nemobot 框架的理论基石。从左到右复杂度递增：Type 1 最简单（查表即可），Type 2 有数学公式保证最优解，Type 3 需要启发式搜索，Type 4 则是最具野心的"能自己学习的机器"。Nemobot 的核心贡献就是展示 LLM 如何增强每一类的表达能力。*

1. **Type 1 - 字典型机器**：预存解决方案的系统
2. **Type 2 - 使用公式的机器**：数学严格可解游戏的系统
3. **Type 3 - 应用原理的机器**：复杂场景下的启发式驱动系统
4. **Type 4 - 学习机器**：通过经验适应的系统

LLMs 从以下方面增强了这些类别：
- 字典型游戏 → **压缩状态-动作映射**
- 严格可解游戏 → **应用符号推理**
- 启发式游戏 → **集成极小极大算法与众包数据**
- 学习型游戏 → **采用带人类反馈的强化学习**

本探索的核心是 **Nemobot**——一个体现这些原则的游戏博弈聊天机器人和交互式编程环境。Nemobot 赋能用户设计和部署游戏博弈智能体，利用 LLM 扩展 Shannon 分类法。通过井字棋、Nim 和 Mancala 等游戏，它展示了 LLM 驱动的 AI 游戏编程进展，集成了自然语言理解和生成能力。

### 关键 Takeaway

- **要点 1：智能 vs 编程的本质差异**。直接 prompt LLM 玩游戏得到的是黑箱行为，不可调试、不可复现。Nemobot 的核心洞察是将 LLM 能力封装为**可检查、可编程的模块化组件**，这是从"AI 玩游戏"到"AI 帮人做游戏 AI"的关键跨越。

- **要点 2：游戏作为世界模型的训练场**。游戏棋盘是离散、可观、规则清晰的微观世界（microworld），为 LLM 学习内部表征提供了理想测试床。作者引用的多项研究表明，Transformer 可以自发形成世界模型的内部表征——这对理解 LLM 的"理解"本质具有重要意义。

- **要点 3：Shannon 四类法的现代化扩展**。本文不是简单回顾历史，而是将 Shannon 1950 年代的分类法逐一对应到 LLM 能力的具体增强方式（压缩、符号推理、众包+minimax、RLHF），形成了完整的理论映射。

- **要点 4：众包作为 AI 训练的数据源**。 Kasparov vs "全世界"（1999）的对局被用作众包决策的原型案例。众包不仅是收集数据，更是让 LLM 暴露于多样化的推理风格和创造性策略——这比单纯的自博弈（self-play）更能提升鲁棒性。

## Section 2 — 大语言模型下的游戏博弈（Game Playing with Large Language Models）

### 中文翻译

### A. 基于 Shannon 分类的 LLM AI 游戏

**Type 1 - 字典型机器（Dictionary-based Machines）**：

Shannon 描述了预定义每种可能的游戏局面并将对应的走法存储在"字典"或函数表中的系统 [2]。传统实现需要大量内存资源，限制了其实用性仅适用于井字棋等简单游戏。在此基础上，我们提出可以利用**语言模型通过从游戏数据中学习模式来增强这种方法**。我们假设这有效地将字典**压缩为一个更小、泛化的模型**，允许系统推断先前未见位置的动作而无需穷举存储。

[扩展] 这里的"压缩"比喻非常精确：传统字典型方法需要存储每个状态的响应（如井字棋的 19,683 种配置各自的最优走法），而 LLM 通过训练学到了状态空间的内在模式，相当于用一个参数化的函数近似了这个巨大的查找表。这本质上是一种**有损但有意义的压缩**——损失了极端情况的精度，但换来了泛化能力和存储效率。

这种方法为新程序员提供了一种实用的学习 AI 游戏编程的方式——通过实时编辑预定义走法，获得 LLM 函数的动手经验，决定何时使用 LLM 或本地内存，并理解交互环境中的战略决策。

**Type 2 - 使用严谨数学公式的机器（Machines using Rigorous Mathematical Formulas）**：

Shannon 概述了能够在 Nim 等数学可解游戏中完美博弈的系统 [27]。在所提出的框架中，**LLM 充当交互式副驾驶（copilot）的角色**，负责用自然语言解释最优走法背后的推理。此设计的目的是让这些系统对 programmer 有价值——programmer 受益于模型表达 Nim 等游戏中最优走法背后逻辑的能力。通过集成 LLM，这些游戏不仅能提供解决方案，还能通过将复杂计算分解为逐步洞察来**提供实时的编程辅助** [30]。这在算法执行与策略制定之间架起了桥梁，使 AI 游戏中以数学证明的策略更加易于理解。

此外，**AI 辅助编程的概念** [31], [32] 和现有 LLM 代码生成能力可用于为这些游戏背后的数学策略创建代码。举例说明，在 1969 年的《欧几里得游戏》（The Game of Euclid）[33] 中，玩家轮流从较大堆中移除较小堆的倍数，目标是把一堆减至零。该游戏有一个与**最大公约数（GCD）计算**相关的最优博弈策略。我们证明 LLM 可以超越分析游戏本身——它能**生成代码来实现最高效的 GCD 算法**，如经典欧几里得算法或其优化变体。例如，给定两堆石头 a 和 b（a > b），LLM 可以生成代码来迭代计算 `a mod b`，识别减少 a 的最优 m（m × b），并确保高效计算同时避免输棋配置。

[扩展] 这个例子非常巧妙地展示了 LLM 在 Type 2 场景中的双重角色：既是策略的解释者（告诉玩家为什么这么走），又是策略的实现者（写出高效算法代码）。这种"解释+实现"的双重能力是纯数学公式所不具备的——传统方法只能给出答案，不能教你怎么得出答案。

**Type 3 - 应用一般博弈原则的机器（Machines employing General Principles of Play）**：

这类机器处理象棋、扑克和桥牌等**不存在精确解但启发式策略指导决策**的游戏 [1]。语言模型可以综合广泛的游戏相关知识、人类策略和专家见解，为各种场景生成**上下文感知的策略**。通过融入蒙特卡洛树搜索等技术并利用自博弈（self-play）（如 AlphaGo [34], [35] 所演示的），LLM 通过在人类和 AI 生成的对局上进行训练来精炼策略。它们的**思维链推理** [36], [14] 允许它们将复杂问题分解为逐步解决方案，建议最优路径的启发式并推荐高效的搜索算法——从而将路径规划和战略探索等计算外包给高效算法。这不仅简化了决策过程以提高游戏性能，还促进了 LLM 与可以测试策略的人类玩家之间的交互 [32]。在编程这些策略游戏时，LLM 可以模拟不同技能水平并通过自博弈解释决策，提高概率推理的效率 [35]。

**Type 4 - 学习机器（Learning Machines）**：

作为 Shannon 分类法的巅峰，学习机器与现代数据驱动方法紧密契合。Shannon 设想的是通过**试错学习**增强策略的机器，基于大量经验持续精炼其方法 [1]。同样，LLM 可以作为**自适应工具**集成，协助通过试错来进行编程和精炼 [37]。LLM 可以通过解释反馈循环和训练数据在学习中的作用，指导**自我改进的 AI 对手**的开发。它们加速了强化学习——特别是 Michie 的 Boxes 算法 [38], [39]，这是一个随机的、试错系统，学习在未知环境中避免失败，是现代强化学习 [37] 的直接知识祖先。通过生成走法、评估结果和精炼策略，LLM 充当**AI 教练**——模拟熟练对手、提供游戏反馈、模仿奖励模型，同时其自然语言洞见促进了**可解释的战略游戏博弈**。

### B. 通过 LLM 函数的可编程提示（Programmable Prompts via LLM Functions）

大语言模型具有从海量（原始无标签）数据集中综合知识的能力，使其成为**自主学习和决策的强大工具** [18]。我们可以设计**不仅仅遵循预编程规则的 LLM 智能体**——它们可以根据遇到的上下文生成策略、响应和解决方案。这与 Shannon 关于机器逐渐编写更多自身指令的愿景一致。

例如，在游戏博弈系统中，不再局限于预定义策略或僵化算法，可以向不同的 LLM 发出查询以**生成可行策略**，以最少的人工输入迭代改进游戏学习性能。

**可编程提示工程**通过允许开发者使用 LLM 函数实现逻辑响应行为进一步增强了这种能力。与其编写详尽的代码覆盖每种可能的情况，**提示工程允许程序通过上下文相关的指令在实时动态理解和响应各种情况**。这提供了一种灵活、可扩展的问题解决方法——机器通过基于提示的指导"编写"自身的指令。LLM 函数因此用于在游戏中生成战略决策，根据先前的结果调整行为，并通过反馈循环精炼策略。

[扩展] 这段话的核心论点是将"提示词"升级为"一等公民（first-class citizen）"的编程构件。传统编程中，if-else 逻辑硬编码在代码里；而在 Nemobot 中，部分逻辑被委托给 LLM 函数——这些函数通过提示词定义，可以被版本控制、测试和共享。这是一种元编程（meta-programming）思想的应用。

因此，不再仅依赖预编程策略，**AI 游戏引擎可以通过解释提示并根据实时反馈自动调整其玩法来自主生成新策略**。这种通过对话数据的机器学习实现的更大自主性的转变 [40] 不仅是一项最新的技术成就，也反映了我们与机器交互方式的深层**哲学转变**——从直接指示它们转向以一种允许它们通过自然语言生成和理解来"编写"自身程序的方式引导它们。

Michie 同时代的 **"memo functions"（记忆函数）或 memoization** [41], [42], [37] 如今被理解为一种**加速学习（speedup learning）形式**——存储和重用之前计算的结果，减少冗余并提高效率。因此，LLM 函数可被视为一种**神经化记忆化（neuralized memoization）**形式，提供更动态、上下文感知的响应。

[扩展] "神经化记忆化"是本文最具洞察力的概念之一。传统 memoization 用 `(input, output)` 键值对缓存结果；LLM 则用其数十亿参数"缓存"了训练数据中的模式——当输入与训练分布中的某个模式相似时，LLM 能"回忆起"相应的输出。这不是精确匹配，而是语义层面的模糊匹配。

事实上，Michie 的 memo 函数 [41], [42] 在大语言模型使用的 **KV 缓存**中找到了直接的现代实例——KV 缓存存储先前计算的注意力键和值以将生成复杂度从二次降低到线性。除了这种性能优化，该概念还可以在三个方向上扩展：

| 扩展方向 | 核心思想 | 类比 |
|----------|----------|------|
| **语义记忆化（Semantic Memoization）** | 通过嵌入相似性而非精确输入匹配进行缓存 | LLM 对"意思相同但表述不同"的查询返回相同响应 |
| **程序记忆化（Procedural Memoization）** | 不仅缓存结果，还跨智能体工作流缓存成功策略或工具使用序列 | 学会的不只是答案，而是解题方法 |
| **分布式记忆化（Distributed Memoization）** | 在多个智能体间共享缓存知识 | 多个 Agent 共享"经验"，集体加速学习 |

因此，将计算视为可学习的资源，从符号缓存扩展到神经化记忆化和多智能体学习，通过启用自适应学习连接到强化学习——过去的交互精炼游戏策略并改善协作问题解决 [37]。

### C. 协作提示工程（Collaborative Prompt Engineering）

有效的提示工程是利用 LLMs 完成游戏博弈等专门任务的关键，但往往需要**实验经验和专业知识** [43], [18]。**众包（Crowd-sourcing）可能提供一种可扩展的解决方案**，利用集体智慧来创建和精炼优化的 LLM 提示以用于战略游戏任务 [44], [45]。确实，1999 年 Garry Kasparov 与"全世界"的国际象棋比赛就是大规模众包的典范——全球参与者结合了人类直觉、计算机引擎和专家建议 [46]。Kasparov 经过 62 步后的险胜凸显了**人机协作中集体决策的潜力和局限性**。众包通过利用多样化参与者视角来精炼响应 [40], [45]，提供了一种可扩展的方式来改善自然语言理解。众包将在智能体工程中发挥重要作用——提供**可扩展的人类监督和自主智能体自身无法生成的多样化训练数据**。

[扩展] Kasparov vs The World (1999) 是计算机科学史上著名的案例。全球互联网用户投票决定走法，配合多台计算机引擎的分析，最终 Kasparov 以微弱优势获胜。这个案例说明了两个关键点：(1) 集体智慧有时可以匹敌甚至超越单个天才；(2) 但协调成本和意见分歧也是巨大挑战。Nemobot 的众包设计正是试图保留前者的优势同时缓解后者的困难。

在战略游戏编程中，众包通过**改进强化学习模型和简化评估**来增强 LLM 微调。它为策略、走法解释、对手模拟和多样化的游戏风格生成提示，提升 AI 适应性。这个协作过程聚合了全面的数据集，发掘了高性能模式，并培养了创新。通过将这些洞见整合到 LLM 智能体（如第三节中的 Nemobot）中，它**规模化人类创造力**，增强人机增强（human-AI augmentation），推动战略 AI 游戏开发中的新编程范式，如图 1 所示。

复杂 AI 游戏的学习过程将涉及大量的试错，但**众包可以通过聚合多样化反馈和人类洞见来减少开销**。可编程提示通过提供一个结构化框架来编码、精炼和评估集体人类专业知识的方式放大了众包效果。通过 LLM 函数，这些提示使智能体能够动态模拟战略决策和问题解决。来自多样化社区的众包将 LLM 智能体暴露于广泛的推理风格和创造性策略，与 Michie 的神经化记忆化相一致。这种方法允许**迭代改进**，结合学习到的策略与可编程模块以随时间提升性能。

### D. AI 游戏编程智能体（AI Game Programming Agent）

为了充分利用 LLM 能力，必须开发具有以下特征的工具：

**特征一：优化计算效率和战略有效性**
- 融合**思维链推理（chain-of-thought reasoning）**，允许开发者探索 AI 智能体的内部工作机制和决策过程
- 促进对策略如何生成和精炼的更深理解——因为 LLM 可以解释最优走法的逻辑并提供试错学习的洞见

**特征二：促进协作和可扩展开发**
- 利用 LLM 函数**生成代码并辅助调试**
- 支持开发者就策略制定和游戏设计进行协作，同时支持多种方法
- 这种可扩展性也鼓励团队改进和迭代策略精炼

**特征三：为不同专长的开发者提供可访问性**
- 提供**自适应框架**供动手探索，配备简化实验的 LLM 驱动工具
- 工具的**分层分类**（基于 Shannon 分类法）结合 LLM 驱动的代码生成，兼顾新手和高级程序员
- 通过促进与 LLM 的动态协作，系统能够实现策略的高效实时生成、优化和适应

最终目标是实现**可扩展的神经化 memoization**，提高算法效率并赋能开发者无缝创新和精炼解决方案。

- **要点 4：AI 游戏编程工具的三维需求**。计算效率×战略有效性（深度）、协作可扩展性（广度）、可访问性（易用性）——这三个维度构成了评价此类工具的完整坐标系。

## Section 3 — Nemobot 游戏编程（Nemobot Game Programming）

### 中文翻译

我们提出 **Nemobot**——一个用于在 Facebook Messenger 和 Telegram 等平台上创建、训练和测试 AI 游戏智能体的编程框架。在本节中，我们提供系统架构概述，详细介绍交互式训练工作流，并描述旨在支持不同专长开发者的**分层课程体系（Tiered Curriculum）**。

### A. Nemobot 概览（Overview of Nemobot）

Nemobot 提供了一个动态且直观的环境，开发者可以在其中为战略游戏设计和训练 AI 智能体，深入了解 AI 的底层机制。如图 3 所示，该平台包含：

1. **编程面板（Coding Pad）**：用于编写游戏逻辑和 AI 推理启发式
2. **聊天游乐场（Chat Playground）**：与 AI 交互以生成和测试数据
3. **分析门户（Analysis Portal）**：评估 AI 性能并精炼启发式

流程从用户设计**双人战略游戏**开始——玩家轮流行动、累积奖励直到游戏结束。系统提供了一个**基于 NodeJS 的模板**来简化此过程，包含可自定义函数来定义：
- 初始状态 $S_0$
- 转移函数 $f(S_{i+1}|S_i, a_i)$（其中 $a_i$ 表示玩家在状态 $S_i$ 下的动作）
- 终止状态 $S_e$

然后用户通过定义**启发式函数 $H_0(a_i|S_i)$** 来编程 AI 智能体——该函数规定了智能体在各种场景中的决策方式。

[扩展] 这个设计的关键在于"启发式"的通用性——它可以是简单的规则（如"随机选择"）、确定性公式（如 Nim-sum 计算），甚至是 LLM 函数调用。这种统一抽象使得同一框架能支持 Shannon 全部四类游戏。

Nemobot 还赋能用户通过利用 **LLM 函数**来融合高级算法设计。用户可以通过在平台内用自然语言指定高级逻辑（如描述和示例）来绕过复杂的编程挑战。这些 LLM 函数（如图 4(c) 部分）无缝集成到主游戏流程中，利用 LLM 的自然语言理解和代码生成能力。这一特性确保了不同编程背景的开发者的可访问性，同时保持了 AI 游戏开发的健壮编程范式。

### B. 系统架构与实现（System Architecture and Implementation）

Nemobot 系统架构的设计注重模块化、低延迟和可扩展性。系统包含三个核心层：

| 核心层 | 设计原则 | 具体实现 |
|--------|----------|----------|
| **模型无关的 AI 集成** | 平台架构独立于任何特定 LLM | 采用**"自带密钥"（BYOK / Bring Your Own Key）**原则，允许用户通过 API 连接各种模型（如 GPT-4、GPT-3.5） |
| **状态管理与延迟** | 确保响应迅速的游戏体验 | 游戏逻辑和状态管理在本地执行环境中处理；活跃游戏期间状态存储在内存中；**仅当用户显式触发"LLM函数"时才查询 LLM** |
| **数据持久化与众包** | 不中断实时交互 | 单个游戏状态是瞬时的；性能指标和训练结果批量异步同步到云端数据库；聚合众包数据（胜率、成功启发式等） |

[扩展] "仅当触发 LLM 函数时才查询 LLM"这个设计决策至关重要。游戏循环本身不需要 LLM 参与——这保证了核心玩法不受 LLM 推理延迟的影响。LLM 只在需要"思考"的时刻被调用（如计算最优走法或解释策略），这是一种精心设计的"懒加载"策略。

### C. 交互式 AI 训练（Interactive AI Training）

AI 智能体的核心工作流涉及基于已定义策略的感知、处理和决策。该策略可以是预编程的确定性规则，也可以通过强化学习等方法迭代学习。为了支持游戏博弈 AI 智能体的开发，Nemobot 提供了一个交互式框架，用户可以从头构建和训练 AI 系统，如算法 1 所述。

**Algorithm 1: 使用 Nemobot 进行交互式 AI 训练**

```
输入: 初始启发式 H₀, 测试数据集 D, 损失阈值 τ
输出: 训练后的启发式 Hₖ

1: H₀ ← 初始启发式, k ← 1, R ← ∅
2: while true do
3:   for i in [1, count(Dₖ)] do
4:     R.add(Rᵢ ← reward(Dₖᵢ))    // 对每个测试用例计算奖励
5:   end for
6:   if loss(R) ≤ τ then           // 如果损失低于阈值则停止
7:     break
8:   end if
9:   Hₖ ← update(Hₖ₋₁, R)         // 基于奖励更新启发式
10:  k ← (k + 1)
11:  record <Hₖ, R>                 // 记录本轮的启发式和奖励
12:  R ← ∅                          // 清空奖励集
13: end while
14: return Hₖ
```

算法 1 展示了一个迭代过程，其中 AI 的决策启发式被优化以提高其性能。在每次迭代中，启发式针对测试用例数据集 D 进行评估，产生指导后续迭代调整的奖励 $R_i$。该循环持续进行，直到 AI 满足预定义的性能阈值 $\tau$。

[扩展] 这个算法的通用性值得注意——`update()` 函数在不同场景下可以有不同的实现：在自动机器学习中可能是基于梯度的优化；在基于规则的 AI 系统中可能代表对决策规则的手动调整；如果启发式是静态的，则完全跳过训练循环。这种灵活性使得 Nemobot 能适应 Shannon 四类游戏的不同需求。

**实际示例 - Nim 游戏启发式**：

用户首先定义一个初始启发式 $H_0$，指定 AI 在给定状态下的动作及相应奖励。例如在 Nim 游戏中，$H_0$ 可能定义 AI 的走法为：

$$n_{\text{AI}} = \max(r, 3 - n_{\text{user}})$$

其中 $r$ 是剩余物品数量，$n_{\text{user}}$ 是对手的上一步动作。Nemobot 利用 **LLM 驱动的代码生成**简化此过程——允许用户用伪代码或自然语言而非可执行代码来描述启发式。这种抽象降低了入门门槛，使用户专注于**逻辑推理而非语法**。

一旦定义了启发式，用户就可以在聊天游乐场中进行测试——动态游戏和数据生成同时发生。AI 以聊天机器人界面部署，从初始状态 $s_0$ 开始，用户与智能体交互，观察其对状态 $s_i$ 的响应动作 $a_i$。最终奖励 $R_k$ 被记录用于分析。为加速测试，Nemobot 支持**自动化游戏博弈**——通过随机动作生成（模拟"众包训练"）或基于启发式的响应（类似"由另一个 AI 训练"）。

启发式更新由性能分析引导。用户利用 LLM 功能迭代精炼 H，调整策略或定义新动作。启发式范围可从确定性动作的精确公式到表示高层策略的抽象规则。此外，Nemobot 支持将**历史训练数据融入 AI 的状态表征**，实现更复杂的决策并与现代自学习方法论保持一致。

### D. 分层课程体系（Tiered Curriculum）

Nemobot 通过提供结构化路径来支持不同专长的开发者，灵感来源于 Claude E. Shannon 的游戏博弈机器分类法 [2]：

| 课程层级 | 对应 Shannon 类型 | 内容描述 |
|----------|-------------------|----------|
| **基础开发（Foundational Development）** | Type 1 - 字典型机器 | 为预定义游戏构建字典型 AI 智能体，为每个状态 $s_i$ 定义动作 $a_i$ |
| **中级规则系统（Intermediate Rule-Based Systems）** | Type 2 - 公式型机器 | 使用 NodeJS 模板创建自定义游戏，构建遵循决策算法 $f(a_i|s_i)$ 的基于规则的 AI |
| **高级数据驱动 AI（Advanced Data-Driven AI）** | Type 3/4 - 启发式/学习型机器 | 利用 LLM 函数设计复杂自适应 AI 系统，过渡到构建具备自我改进能力的学习型 AI |

[扩展] 这种分层设计的核心教学理念是**脚手架式学习（scaffolded learning）**——学生不是一开始就面对最复杂的强化学习任务，而是从最简单的查表法开始，逐步建立直觉后再引入更抽象的概念。每一步都建立在前一步的理解之上。

这种结构化进阶以**启发式驱动决策为基础**，逐渐融入数据驱动方法论。通过简化初始开发并强调迭代增强，Nemobot 确保平滑过渡到高级 AI 训练技术。通过智能体工程——自主 LLM 智能体规划、执行和精炼学习策略——系统可以**动态适应个别学习者的轨迹**，选择适当的任务和反馈机制而无需持续的人类监督。

- **要点 2："门控接口"的安全考量**。作者明确意识到生成式 AI 工具的风险，设计了中间层来管理用户与 LLM 的交互。这反映了负责任的 AI 设计理念——不是禁止使用，而是通过结构化方式安全使用。

### A. 字典型游戏：井字棋（Tic Tac Toe）

井字棋是 Shannon 第一类游戏博弈机器的理想示例——使用存储所有可能游戏状态**预计算解决方案的字典型系统**。其状态空间可控（19,683 种配置），每个状态配对一个最优走法。这种确定性方法确保每个游戏状态都有明确定义的最优响应（如最大化获胜机会或强制平局）。虽然有效，但这种方法可能在基于规则的 AI 游戏中内存密集且实现复杂。

[扩展] 19,683 = $3^9$（9 个格子，每个格子可以是 X、O 或空）。虽然数字看起来不大，但手动编码每种状态的最优响应既枯燥又容易出错。这正是 LLM "压缩字典"的价值所在——不需要穷举存储，而是学到了"占据中心""阻止对手连线"等高阶模式。

为了解决这些限制，我们采用 **LLM 游戏智能体**——它在无需显式存储大量状态-动作对或创建复杂的嵌套条件循环的情况下复制了字典型系统的行为。用户通过与 Nemobot 交互来进行走法，LLM 作为对手响应，维持字典型系统的**确定性特征**。为便于交互，3×3 棋盘用 0-8 数字标记单元格，对应棋盘位置。用户通过指定想要放置标记的单元格来输入走法（如"0"表示左上角，"4"表示中心）。

**与 LLM 智能体通信时，游戏状态被序列化为结构化文本格式**——捕获 X、O 和空单元的位置。这种结构化输入有效地以智能体可处理和响应的方式描述了游戏的当前状态。例如，假设 LLM 智能体已经知道其在 Nemobot 配置中嵌入的角色和规则，给定状态的提示词可以这样写：

**示例提示词：井字棋策略**
```
当前配置：单元格 0（左上角）被 'X' 占据。
          单元格 4（中心）被 'O' 占据。
          其余所有单元格为空。
目标：分析棋盘状态并执行玩家 'O' 的最优走法以防止失败或确保胜利。
```

[扩展] 这种结构化提示词设计非常关键——它将视觉化的棋盘状态转换为 LLM 能理解的文本描述。注意提示词不仅包含事实信息（哪些格子被占了），还包含指令（"分析...并执行最优走法"），这引导 LLM 进入"推理模式"而非单纯的"识别模式"。

**不是预计算所有可能的走法并将它们存储在字典中，LLM 游戏智能体利用其预训练知识和上下文理解来动态生成最优走法**。后端通过 API 将此查询发送给 LLM，LLM 根据当前配置响应最具战略性的走法。这种动态查询过程消除了穷举存储的需求，有效地将"字典"压缩进了 LLM 游戏智能体的推理能力中。对于任何给定游戏状态，Nemobot 确保智能体的响应是一致且确定性的，遵守传统字典型机器的基础原则。

尽管 LLM 游戏智能体动态生成响应，但这些响应**与井字棋的既定最优策略一致**——其决策建立在预训练知识之上。这种设计确保了选择最优走法所需的基础知识**一次性嵌入 LLM**，使其甚至能推断出先前未见位置的最佳走法。与传统方法相比，该方法显著减少了内存使用和计算开销——传统方法依赖穷举搜索或对每个决策评估嵌套条件循环。通过利用 LLM 游戏智能体的泛化和解释游戏规则的能力，Nemobot 实现了**增强的可扩展性和适应性**，在无需重复基于规则评估的情况下提供高效准确的游戏体验。

### B. 使用严谨数学公式的机器：Nim 游戏

Nim 是经典的组合博弈游戏，属于 Shannon 第二类游戏博弈机器——**使用数学公式的系统**。其最优策略基于 **Nim-sum（尼姆和）**——通过异或运算判断必胜/必败位置。在 Nemobot 中，被迫取走最后一个物品的玩家**输**。AI 在每一回合计算 Nim-sum，使用公式确定最佳走法：非零 Nim-sum 表示必胜位置，零 Nim-sum 表示必败位置。LLM 将 Nim-sum 计算翻译成自然语言，提供关于最优走法的洞见，使游戏更具教育性和可访问性。

[扩展] 这里采用了"取最后一个输"的变体（misère play），比标准版（取最后一个赢）更复杂但更有趣。LLM 的角色不仅是计算 Nim-sum，更是将数学概念翻译成人类语言——这是纯算法无法做到的。

游戏逻辑将棋盘当前状态序列化为结构化文本格式，详细说明剩余石子数量以及轮到谁走。序列化状态随后传给 LLM——LLM 利用其预训练的 Nim 游戏数学解法知识来确定最优走法并以自然语言呈现数学策略。例如，假设剩余 8 个石子，LLM 可能收到以下提示：

**示例提示词：Nim 游戏逻辑**
```
游戏状态：只剩一堆，包含 8 个石子。
约束：你被允许移除 1、2 或 3 个石子。取走最后一个石子导致失败。
任务：应用获胜数学策略（Nim-sum 分析）精确计算本回合应移除的石子数。
      为你的决定提供简要理由。
```

### C. 应用一般博弈原则的机器：Mancala 游戏

Mancala 是一个双人策略游戏，代表了 Shannon 第三类游戏博弈机器——**依赖一般博弈原则的系统**。其复杂的决策空间使得**预计算最优走法不切实际**，需要启发式策略和博弈树来引导决策。然而，由于不断扩大的决策空间，这种方法资源消耗大。Nemobot 通过使用 **LLM 函数动态查询建议**来解决这些挑战——在不替换博弈树的情况下增强博弈树策略。LLM 分析当前游戏状态以检索相关策略，减少计算和内存需求，同时保留博弈树作为决策基础。

[扩展] Mancala（非洲宝石棋）的决策空间远大于井字棋和 Nim——每步有 6 种合法选择，且游戏长度可达 40-50 回合。完整的博弈树搜索在此不现实。Nemobot 的方案是不抛弃博弈树，而是用它作为"骨架"，用 LLM 提供"肌肉"——即对节点的智能评估。

游戏逻辑首先将当前棋盘状态转换为结构化文本，详细说明每个坑（pit）中的种子数、商店（store）值和活跃玩家。Mancala 棋盘按数字标记：
- 底行坑 0-5（从左到右）
- 顶行坑 7-12（从右到左）
- 商店分别在位置 6（底行）和 13（顶行）

**示例提示词：Mancala 博弈树分析**
```
状态表示：
对手（底行）：坑0有4颗种子，坑1有1颗种子，坑2有3颗种子。商店：8。
智能体（顶行）：坑7有2颗种子，坑8有0颗种子，坑9有4颗种子。商店：6。
指令：利用博弈树分析确定从此位置获得最高战略优势的走法。
```

![Figure 5: Mancala 游戏的聊天机器人界面](assets/page-005-img-01.png)

*Figure 5: Mancala 游戏的聊天机器人界面。[扩展] 这张图展示了 Mancala 游戏在 Messenger/Telegram 等 IM 平台上的实际运行效果——棋盘用 emoji 或文字渲染，玩家通过发送消息来选择走法。这种"聊天即游戏"的交互形式大大降低了玩家的使用门槛。*

Nemobot 利用 LLM 函数在**预计算的博弈树中查询最优路径**——根据当前游戏状态动态检索策略。例如在 Mancala 中，如果底行玩家在坑 0 有 3 颗种子、坑 1 有 2 颗、坑 2 有 4 颗，而顶行玩家在坑 7 有 1 颗、坑 8 有 0 颗、坑 9 有 5 颗，LLM 会识别最佳走法（如从坑 2 下子以获得额外一回合）。程序员可以获得 **LLM 从相似场景中概括出的不同走法选择**——这与传统的运行极小极大算法加博弈树的方法 [1] 不同。

### D. 通过强化学习的学习机器（Learning Machines by Reinforcement Learning）

Shannon 的第四类——**学习机器**——通过经验改进策略，使用试错、强化学习和自我改进。在 Nim 和 Mancala 等游戏中，这些系统超越预定义规则，利用自适应算法和对弈数据。有效的模拟对于收集高质量训练数据至关重要——包括所需试验次数的洞察和数据准确性保障。**用此数据微调 LLM 函数增强了决策制定**，使 AI 更高效和更具适应性。

**Nim 案例**：开发了基于 Web 的浏览器模拟来生成广泛的对弈数据。模拟允许玩家与各种初始设置交互（不同的堆大小和每回合可移除的石子数规则）。利用此平台，AI 收集最优策略和走法序列数据。收集到的对弈数据随后通过提示工程用于**微调 LLM 函数**。例如，AI 学会推荐基于最大化获胜概率的走法，通过对策略的微调理解动态适应游戏场景。Web 模拟演示地址：**Nim Simulation**。

**途径一：Michie's Boxes 算法（自博弈）**
模拟利用 Donald Michie 的 Boxes 算法 [38] 通过自博弈进行。走法基于当前状态的"盒子"概率性地选择——**获胜走法增加重选概率，失败走法降低概率**。经过多次迭代，这个试错过程在多样化场景中发现最优策略，且无需专家示例。

[扩展] Michie's Boxes（1968）是强化学习的鼻祖之一。每个状态是一个"盒子"，盒子里的每个可选动作有一个计数器（或概率）。选了之后赢了就加分（更可能再选），输了就扣分（更不可能再选）。这本质上就是 Q-learning 的前身——只是用频率计数代替了价值函数估计。

**途径二：众包（Crowdsourcing）**
人类玩家通过与使用 Shannon 经典极小极大算法 [1] 的智能体对弈来贡献多样化的走法。这将 LLM 智能体暴露于更广泛的战术和策略，提高其对不同游戏风格的适应性。此外，LLM 智能体充当**教练（instructor）**——分析对弈、对错误提供实时反馈、建议最优策略。这通过突出次优走法和引导探索更好的替代方案来加速学习，培养更具战略性、长期的决策制定。

**评估方法——众包排行榜（Leaderboard）**：

为实现上述方法的性能评估，我们实施了**众包排行榜**——人类玩家在此与 Nemobot 竞争。排行榜地址：**Mancala Leaderboard**。该排行榜捕获人机对弈数据——AI 使用经典极小极大算法配合三种不同树深度（对应简单、中等、困难难度级别）。收集的数据随后供 LLM 用于精炼其决策（确定最优走法或生成自博弈算法）。排行榜记录玩家分数并提供关于人类玩家所用策略的有价值洞见。这种多样化输入确保 LLM 在 LLM 函数编程中整合了广泛的提示词，最终增强其适应性和性能。

**示例提示词：战略批评与响应**
```
状态分析：
玩家（底部）：坑0和坑1各含2颗种子。商店：0。
AI（顶部）：坑2和坑3各含2颗种子。商店：0。
任务是AI的回合。识别最优走法序列。此外，批评一个潜在次优走法以展示战略错误。

示例响应：
建议：从坑2开始下子。
策略：此动作将种子分发给坑3和坑0，落入商店触发"免费回合"机制。
      在下一回合，AI应从坑3下子，从而俘获坑0中对手的种子。
批评：立即从坑3开始是劣质的——它放弃了免费回合且未能利用俘获机会。
     优先考虑延长回合序列的动作对于最大化得分至关重要。
```

### 关键 Takeaway

- **要点 1：四类游戏的完整实现路径**。从井字棋的字典压缩 → Nim 的数学推理+代码生成 → Mancala 的博弈树+LLM查询 → 强化学习双重途径（Michie Boxes + 众包）。每一类都有具体的提示词示例和部署方案。

- **要点 2："战略批评与响应"模式的教育价值**。LLM 不仅给出最佳走法，还**主动批评次优选择并解释原因**。这种"正例+反例"的教学方式比单纯给答案有效得多。

- **要点 3：众包排行榜的双重作用**。既是激励机制（玩家想上榜），又是数据源（收集人类策略多样性）。这巧妙地将"游戏乐趣"与"AI 训练需求"对齐。

- **要点 4：真实教育场景的规模化验证**。251+80+30 = 360+ 名学生跨越三所顶尖大学、三年时间、从 Codex 到 ChatGPT 的工具演进——这是罕见的长时间、大规模教育实验。

### A. 文献定位与影响（Implications and Positioning in the Literature）

为了将本工作置于基于 LLM 的游戏 AI 更广阔图景中进行比较性定位，将 Nemobot 与几个著名系统进行比较是有益的：

| 对比系统 | 核心特点 | 与 Nemobot 的区别 |
|----------|----------|-------------------|
| **Voyager** [48] | 使用 GPT-4 在 Minecraft 中自主探索和获取技能的开端具身智能体 | 完全自主但无机制让人类程序员检查/修改/分享其发现的策略 |
| **ChessGPT** [8] | 架起国际象棋的策略学习与语言建模桥梁 | 专用于单一领域，不暴露可编程接口供教育使用 |
| **Generative Agents** [49] | 通过 LLM 驱动的记忆和反射模拟可信的人类行为 | 关注社会模拟而非战略游戏编程 |
| **Suspicion-Agent** [10] | 将心智理论推理与 GPT-4 集成用于不完美信息游戏 | 展示了复杂的多智能体交互但没有教育或编程导向框架 |

[扩展] 这个对比表格非常清晰地界定了 Nemobot 的独特定位——它不是最强的游戏 AI（不如 Voyager 自主），不是最深的专业系统（不如 ChessGPT 在国际象棋领域精深），但它是一个**可编程、可教育、可检查的框架**。这个定位与 GitHub Copilot 在编程领域的定位类似——不是替代程序员，而是赋能程序员。

**相比之下，Nemobot 占据了一个独特的细分市场**：它是一个**可编程的、教育性的框架**，在 Shannon 分类法中将 LLM 函数作为透明、模块化的组件暴露出来。用户不仅消费 AI 生成的游戏玩法——他们通过**协作提示工程和众包反馈主动设计、测试和精炼游戏博弈策略**。这种可编程性将 Nemobot 与那些将 LLM 视为黑箱玩家的系统区分开来，并与**AI 辅助编程** [31], [26], [21] 的更广阔愿景保持一致——其中人类监督和迭代精炼是开发过程的中心。

通过将其设计建立在 Shannon 经典框架的基础上并用现代 LLM 能力进行扩展，Nemobot 架起了**基础 AI 理论与当代实践之间的桥梁**，既提供研究价值也提供教育效用。

最后，本文的算法框架自然延伸到游戏博弈之外——延伸到教育 AI 中的**知识追踪（Knowledge Tracing）和学习如何学习（Learning-to-Learn）** [50]。在知识追踪中，试错机制（如 Michie's Boxes 算法）可以建模学生对概念的逐渐掌握程度：
- 每个知识组件成为一个"盒子"
- 学生的响应（正确或错误）作为奖励或惩罚
- 选择特定策略的概率相应更新
- 这使得无需专家标记技能模型即可实时预测未来表现

与此同时，众包与**学习如何学习**保持一致——通过聚合来自多样化学习者的解决方案策略，允许智能导学系统元学习（meta-learn）哪种教学方法最适合哪种学生类型。经过训练的 AI 智能体作为翻转课堂模型中的**教练**可以成为**元学习顾问**——它分析学生的错误模式，生成个性化练习题，建议替代解题途径 [51]。这确保了任务的复杂性有效地随学习者的熟练度扩展。这些机制共同可以将战略游戏博弈 AI 智能体转变为**参与学习如何学习的自适应教育系统**——通过与未知环境交互不断精炼其预测和战略教学方法，从而随着时间的推移获取关于每个个体如何学习的**元知识（meta-knowledge）**。

[扩展] 这段论述将 Nemobot 的价值从"教人做游戏 AI"升华为"构建自适应教育系统的方法论"。知识追踪→个性化教学→元学习顾问——这条线展示了游戏作为教育载体的深层潜力。

### B. 开放问题与未来方向（Open Issues and Future Directions）

未来研究将把 Nemobot 扩展到多媒体生成和编程等多样化的 AI 任务，丰富 AI 教育。受 Michie 关于**发现即协作（discovery as collaboration）** [42] 思想的启发，未来工作可能开发**游戏过程中实时策略精炼的工具**，结合自适应半监督学习和微调来实现三方协作——LLM 驱动的副驾驶、人类程序员和 Turing 的儿童机器 [42]（即本地较弱的 LLM）。

以下列出了开发 LLM 驱动的游戏博弈智能体和协作可编程提示工程中仍存在的几个开放问题：

**① 长期策略 vs 短期决策**

LLMs 擅长短期决策，但在国际象棋和 Mancala 等复杂游戏所需的长期策略方面面临挑战，需要**记忆机制或外部模型**。未来研究可以探索受 Shannon 层级 [2] 启发的**分层系统**——其中学习通过抽象推进。在最高层，编程副驾驶可以利用**谓词逻辑（predicate logic）**进行高级规划，使用归纳逻辑编程从示例和背景知识中推导出谓词 [52]。

**② 泛化 vs 特化的平衡**

虽然 LLM 在海量数据集上训练以有效泛化，但将其应用于特定游戏领域通常需要大量微调。开发**能够根据游戏反馈进行实时调整的自适应微调方法**，可以使 LLM 在不同游戏类型或环境之间无缝过渡。

**③ 人类反馈的可扩展性问题**

依赖人类反馈进行微调带来了可扩展性挑战。虽然众包反馈可以加速学习，但确保此输入的质量和一致性仍然是瓶颈。未来工作可以探索**自动反馈系统**或**结合人类监督与 AI 生成评估的混合方法**。

**④ 对抗性机器人的伦理挑战**

关于对抗性机器人 [53], [54] 的最新研究表明，虽然它们能够击败顶级围棋 AI（如 AlphaGo [9]），但其被训练来利用**隐藏漏洞而非发展全面的策略**。因此，**平衡竞争性能与伦理设计**是一个挑战。开发者必须确保机器人公平透明地参与博弈，避免利用人类弱点，培养具有挑战性但愉快的游戏体验。随着 AI 系统的进步，用大数据实现这种平衡至关重要——这对游戏中的人机交互提出了重要的伦理考量 [55]。

**⑤ 计算效率 vs 可复现性的权衡**

对于利用大数据的实时交互游戏而言，**平衡计算效率和可复现性是一个关键挑战**。LLM 高度资源密集，使其大规模部署的计算成本高昂。为此，研究**轻量高效 LLM 架构和混合系统**至关重要——这些系统可以将较简单的任务卸载到较小高效的模型，同时保留 LLM 的完整能力给需要更高复杂性的任务。实现这一平衡不仅可以增强可扩展性和降低成本，还能通过在各种应用中实现性能、资源利用和响应准确性的一致基准来确保可复现性 [55]。

## Section 6 — 结论（Conclusion）

### 中文翻译

总之，本文介绍了一种**战略性 AI 游戏编程的新方法**，灵感来自 Shannon 关于自编程自动化的愿景以及 Minsky 将编程视为精炼想法之工具的观点。其核心是 **Nemobot**——一个允许用户创建和部署游戏智能体的交互式平台。通过集成 LLMs，Nemobot 超越了静态指令——使 AI 智能体能够通过内在世界模型、众包洞见和迭代学习策略动态适应复杂的游戏玩法。

这一编程范式展示了 Nemobot 中的游戏如何超越僵化的算法——在 AI 游戏中实现识别模式和演化策略的灵活性与创造性。通过**自主精炼其指令并利用众包数据策选有效提示词**，Nemobot 可以探索复杂的游戏挑战以进行竞争性和战略性博弈。我们的方法与 Shannon 关于**通过学习精炼策略的自编程机器**的愿景保持一致。Nemobot 架起了 AI 理论与 LLMs 之间的桥梁，提供**教育价值和动态游戏设计的新可能性**。

## 全文总结

### 核心贡献

Nemobot Games 是一个**将 Claude Shannon 经典游戏机器分类法与大语言模型能力相结合的 AI 游戏编程框架**。它的核心创新在于：

1. **理论层面**：将 Shannon 四类游戏机器逐一映射到 LLM 的具体增强方式（压缩、符号推理、众包+minimax、RLHF）
2. **系统层面**：提供完整的 IDE 式环境（Coding Pad + Chat Playground + Analysis Portal）支持游戏 AI 的全生命周期开发
3. **教育层面**：设计了对应 Shannon 四类的分层课程体系，在三所大学完成规模化教学验证
4. **概念层面**：提出"神经化 memoization"等新概念，将经典缓存思想推广到 LLM 领域

### 方法论创新

| 创新点 | 传统做法 | Nemobot 做法 |
|--------|---------|-------------|
| 字典存储 | 穷举所有状态-动作对 | LLM 参数化压缩 |
| 数学策略 | 只给答案 | LLM 解释+代码生成 |
| 启发式搜索 | 纯算法 | LLM 查询增强博弈树 |
| 试错学习 | 单一自博弈 | Michie Boxes + 众包双轨 |
| 提示词工程 | 个人摸索 | 协作众包+版本管理 |

### 局限性与未来方向

- LLM 长期策略能力不足 → 需要分层系统和外部记忆
- 泛化-特化矛盾 → 需要自适应微调
- 人类反馈瓶颈 → 需要自动评估系统
- 伦理风险（对抗性机器人）→ 需要公平性约束
- 计算成本高 → 需要轻量混合架构
