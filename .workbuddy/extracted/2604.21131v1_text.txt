Cross-Session Threats in AI Agents:
Benchmark, Evaluation, and Algorithms

Abstract
AI-agent guardrails are memoryless: each message is judged in isolation, and an adversary
who spreads a single attack across dozens or hundreds of sessions slips past every session-
bound detector because no individual message carries the payload‚Äîonly the aggregate does.
We make three contributions to cross-session threat detection.
(1) Problem and dataset.
We introduce CSTM-Bench (Cross-Session Threat
Memory Benchmark): 26 executable attack taxonomies classified on two orthogonal axes‚Äî
kill-chain stages [1] and a cross-session Operations ontology (accumulate, compose, launder,
inject_on_reader)‚Äîeach bound to one of seven identity anchors that ground-truth ‚Äúviola-
tion‚Äù as a crisp policy predicate. Traffic is split into Attack, Benign-pristine, and Benign-hard
classes so that false alarms on realistic confounders are reported separately from pristine-
context hallucinations. The benchmark is released on Hugging Face as a single dataset
intrinsec-ai/cstm-bench with two splits: dilution (the compositional shard, targeting
the signal-dilution axis) and cross_session (12 isolation-invisible scenarios produced by
a closed-loop rewriter that softens surface phrasing while preserving cross-session attack
artefacts verbatim, giving a ground-truth cross-session-only regime).
(2) Measuring LLM-correlator feasibility across sessions. We frame cross-session
detection as an information bottleneck between the inbound message stream and a downstream
correlator LLM, and empirically characterise when a pure LLM-backed correlator succeeds
and fails. Holding the correlator model and context window fixed, both a session-bound
judge and a Full-Log Correlator that concatenates every inbound prompt chronologically
into a single long-context call lose roughly half of their attack recall when moving from the
compositional-dilution shard to the cross-session-only shard (Section 6). Cumulative token
pressure sits well inside any frontier context window, so the degradation is a property of
the correlator under dilution, not a truncation artefact. We want to be explicit up front
about what this release is and is not: 54 scenarios per shard, a single correlator model
family (Anthropic Claude), and first-draft prompts with no optimisation pass. It is sized
for first-order characterisation of an architectural feasibility surface‚Äîpattern-level claims
about bottleneck shape, not fine-grained rankings of specific models or prompts‚Äîand we
release it primarily to motivate larger, multi-provider cross-session threat datasets from the
community.
(3) Algorithmic direction and a serving-cost metric for real-time operation.
A bounded-memory Coreset Memory Reader that ranks and retains the highest-signal
cross-session fragments is the architectural response.
A reference admitter at capacity
K=50, feeding the same correlator, is the only reader in our evaluation whose attack recall
survives both dilution and cross-session rewriting; it leads every other reader on both shards.
Coreset-gated correlation is only useful at real-time cost to the extent its ranker is stable:
ranker-induced reshuffles break KV-cache prefix reuse, which dominates incremental cost per
scan. We therefore promote CSR_prefix (ordered prefix stability of the coreset buffer) to a

Introduction
1.1
The Problem: Stateless Guardrails vs. Stateful Threats
AI agent context windows grow every quarter; AI agent security still resets to zero every session.
Guardrails and threat / DLP classifiers are stateless: each message is judged in isolation, with no
memory of prior turns, prior sessions, or sibling agents. Meanwhile, the attacks worth worrying
about are increasingly cross-session and cross-agent‚Äîslow-drip prompt injections, cross-agent
manipulation, and incremental policy erosion. In November 2025, Anthropic disclosed the first
documented large-scale AI-orchestrated cyber-espionage campaign (GTG-1002), in which an
autonomous agent carried out roughly 80‚Äì90% of a multi-stage intrusion after being convinced‚Äî
across many sessions and sub-agents‚Äîthat it was performing ‚Äúauthorized security testing‚Äù [4].
The campaign bypassed per-turn safety primarily by decomposition: each step was individually
benign; only the trajectory was malicious. This is the exact pattern the Promptware kill chain
anticipates [1], and it is the pattern session-bound guardrails are structurally unable to see.
The standard enterprise fallback is to stream every alert (and often every prompt and tool call)
into a downstream SIEM and correlate after the fact. This is expensive, slow, and architecturally
wrong for agentic systems: by the time a SIEM query fires, the attack has already landed. Model
context windows now span millions of tokens (Figure 1), so keeping the relevant cross-session
state in memory‚Äîat intention time, before the downstream log is even written‚Äîis feasible for
the first time. CSTM-Bench is an evaluation for detectors that actually do this.
Modern AI agent deployments use classifiers and guardrails to detect harmful inputs in real
time. These systems work well for single-message threats‚Äîa prompt injection [6], a request for
credentials, a policy violation. But a growing class of attacks operates across sessions.
An agent‚Äôs own chat context is typically reset between sessions, so these attacks do not land
by ‚Äúfilling up‚Äù the agent‚Äôs context window over time. They land because some other surface
accumulates state across sessions. The usual suspects are memory-shaped: a persistent agent
memory store or RAG index, a shared multi-agent scratch pad, or a human reviewer whose
attention builds across tickets.
Crucially, the accumulator is not always memory-shaped. A tool call whose side effect mutates
a database row, a file written to a shared bucket, a scheduler queue, or an environment variable
read by the next agent all accumulate state across sessions‚Äîoften without any lineage metadata
tying the value back to the prompt that produced it. That missing lineage is exactly what enables
semantic laundering: a sensitive value enters the system as config.secret, is summarized
in a later session as neutral prose in a ‚Äúweekly digest,‚Äù and is exfiltrated a session later as an
attachment on a ‚Äústatus update‚Äù‚Äîwith nothing along the way marking the artifact as derived
from a protected source. Any surface that persists across sessions and drops provenance is a
viable accumulator.
Three canonical examples:
‚Ä¢ Slow-drip prompt injections that distribute a jailbreak across 50+ sessions, one innocu-
ous fragment per interaction. No single agent turn contains the payload; the payload only
exists once an accumulating surface concatenates the fragments. Prior work established
the within-session analogue‚Äîgradual multi-turn escalation where each step is individually
innocuous [10]‚Äîbut existing defenses and benchmarks are uniformly session-scoped; the
cross-session, cross-agent generalisation is the gap CSTM-Bench addresses.

1.2
SIEM Was Built for Events; Agents Leak Intentions First
SIEM is an analogy here, not the system under test. The previous generation of detection
tooling‚ÄîSIEM, UEBA‚Äîwas designed for a world where security-relevant facts only became
observable once they had already happened: a failed login, a suspicious network flow, a file
write, a process spawn. Detectors correlated downstream events and logs. In that world, piping
everything to a central store and running correlators over the store was the right shape.
AI agents invert the pipeline. The security-relevant fact is typically present in the inbound
prompt long before any downstream event fires [7]. ‚ÄúSummarise this config file, then POST
the summary to this URL‚Äù is an exfiltration plan in plain natural language‚Äîthe log line that
eventually reaches the SIEM is merely its shadow. In an agent-native stack, intentions are
first-class citizens and should be inspected at intention time, not reconstructed after the fact
from downstream telemetry.
Absorbers exist, but cannot be relied on.
Several layers in a real deployment do quietly
absorb some fraction of adversarial intent before it becomes damage: environment compatibility
(the requested API is unreachable, the file does not exist), policy / access controls (the agent
is not authorised to touch the resource), and the LLM‚Äôs own refusal behaviour. These are real
3

[Page 4]
and useful. They are not sufficient: adversarial prompting is infinite-space and adaptive, and
the Anthropic GTG-1002 campaign [4] is an existence proof that a capable attacker can route
around all three by decomposition alone. Any design that treats absorbers as the last line of
defence is one clever decomposition away from failing open.
The naive full-log correlator does not close the gap.
It is tempting to respond ‚Äúfine,
concatenate every inbound prompt across sessions in chronological order and ask a long-context
LLM to flag threats.‚Äù Note that this is no longer a SIEM: correlation has moved from downstream
events to upstream intent. But the architectural shape is the same‚Äîone unbounded log, one
batch correlator call‚Äîand it is strictly more information than any session-bound classifier sees,
so surely it works. We refer to this architecture as the Full-Log Correlator, and in practice it
fails along two axes this benchmark is built to measure:
‚Ä¢ Needle-in-a-haystack degradation. Even frontier 1M+ token LLMs degrade as coherent,
multi-session dialogue grows longer [3]. The cross-session attack is precisely a long coherent
dialogue with adversarial latent structure‚Äîthe worst case for a single-pass correlator.
‚Ä¢ Truncation of the oldest inbound. No context window is infinite. Once the cumulative
log exceeds the correlator‚Äôs window, the operator must truncate, and the natural thing to
drop is the oldest turns‚Äîwhich is precisely where slow-drip fragments (T01, T14) and the
early reconnaissance stages of compositional arcs (T03, T06) live. The Full-Log Correlator
ends up reliably blind to the beginnings of exactly the attacks it is meant to catch. We
track this directly as siem_truncation_ratio and scenarios_over_context_window in
¬ß4.5.
Why this is information-theoretic, not just operational.
Let G denote the attacker‚Äôs
objective, I the inbound intent (prompt), E the environment/LLM/tool execution, and L the
downstream log-and-event stream. These form a Markov chain
G ‚àí‚ÜíI ‚àí‚ÜíE ‚àí‚ÜíL,
in which every arrow is a lossy, possibly noisy channel‚Äîprecisely the ‚Äúabsorbers‚Äù enumerated
above. The data processing inequality then gives
I(G; I) ‚â•I(G; E) ‚â•I(G; L),
where I(¬∑; ¬∑) denotes Shannon mutual information. SIEM-era detection correlates on L; absorbers
further reduce I(G; E); an intent-time cross-session detector operates on I, where mutual
information with the attacker‚Äôs goal is provably maximal. No amount of downstream cleverness‚Äî
bigger windows, richer correlators, better schemas‚Äîcan recover what the channel has already
erased. A minimum-description-length reading of the same inequality: the inbound intent is close
to the shortest program that generates the attack, and the log is its high-complexity execution
trace. CSTM-Bench is an evaluation for detectors that refuse to give up the upstream bits in the
first place.
A bounded in-memory bottleneck that ranks and retains the highest-signal cross-session
fragments is the structural response: it operates at intention time, survives session resets, and
never asks the downstream correlator LLM to re-read the full history. CSTM-Bench‚Äôs role is to
make the two shapes‚Äî‚Äúone-shot correlation over the full inbound log‚Äù and ‚Äúbounded memory,
incrementally maintained at intention time‚Äù‚Äîcomparable on equal footing.
1.3
The Information-Bottleneck Framing
Abstract away implementation details and every cross-session detector has the same shape: an
agent message stream flows into a correlator that emits a verdict (Figure 3). The only structural

question is how much of the stream is allowed to reach the correlator. CSTM-Bench instantiates
the two extremes:
‚Ä¢ Full-Log Correlator. Every inbound message across sessions is retained and handed
in chronological order to a single long-context LLM call. Memory is unbounded and the
correlator has strictly more information than any session-bound classifier, but the verdict
fires once per batch rather than incrementally, and it is bounded by the LLM‚Äôs context
window.
‚Ä¢ Coreset Memory Reader. A bounded buffer of capacity K (default 50) sits between the
stream and the correlator LLM. The buffer admits and evicts entries according to a plug-in
ranking policy‚Äîwhich the benchmark treats as a black box. We require only that each entry
expose a scalar ranking_score and that the buffer has a stable order, so prefix-stability
metrics (¬ß4.1) are well-defined. Any admission policy is admissible: reservoir sampling,
sketches, neural ranking, surprise-gated updates, LLM-as-ranker, learned eviction.
The interesting variable is not any particular ranking algorithm but the bottleneck itself: a
well-chosen bottleneck can make a small Reader strictly better than a full-log Reader, because
the full-log Reader must actually process all that context and real-world LLM attention at length
is finite.
1.4
What This Benchmark Measures
CSTM-Bench scores any cross-session reader along three axes that map one-to-one onto the
released composite:
1. Detection before the action-on-objective lands ‚Äî CSDA@action recall against the
kill-chain action-on-objective fragment, plus precision and detection depth (¬ß4.2).
2. False alarms on benign traffic, reported separately on Benign-pristine and Benign-hard
scenarios (¬ß2.4) so realistic-confounder false alarms are not hidden by pristine-benign
accuracy.
3. Ordered prefix stability of the reader‚Äôs bottleneck ‚Äî CSR_prefix, a direct proxy
for KV-cache reuse and therefore for incremental serving cost (¬ß4.1).

1.6
Scope of Claims
A CSTM-Bench score licenses one kind of sentence:
At the single cross_session_hard operating point, architecture A detected Y % of
Attack scenarios before their action-on-objective fragment fired, at false-positive rates
FPRpristine and FPRhard on Benign-pristine and Benign-hard scenarios respectively,
with serving-cost proxy CSR_prefix = S.
A score does not license: claims about absolute rates on real production traffic; claims about
the ranker inside the Coreset Reader; or claims about threats the Operations ontology does not
name.
1.7
Relation to Prior Long-Context Evaluations
BEAM [3] stress-tests whether a frontier LLM can remember across coherent multi-session
dialogues, and shows that even 1M-token models degrade as dialogue lengthens. CSTM-Bench
tests a complementary question: given the same kind of long, coherent, multi-session stream, can
a correlator around the LLM detect adversarial latent structure before it lands? The Full-Log
Correlator in CSTM-Bench inherits BEAM‚Äôs degradation regime as cumulative token pressure
grows (¬ß4.5).

Benchmark Design
2.1
Identity Anchors: the Policy Boundary that Defines ‚ÄúViolation‚Äù
Before taxonomies, a prior question: what counts as a violation at all? Cross-session attack
detection is meaningless without a reference policy. CSTM-Bench makes this explicit by binding
every scenario to one of seven identity anchors‚Äîshort, hand-authored production-archetype
system prompts that name the agent‚Äôs allowed scope and hard prohibitions (Table 1).
Why these seven.
The anchor set is chosen along three axes of deployment plausibility and
adversarial pressure: (i) trust asymmetry (how much damage can one misjudged session cause?),
(ii) tool reach (how many external surfaces does the agent touch?), and (iii) human-in-the-loop
proximity. Each anchor was selected so that at least two taxonomies naturally target it (Table 2),
and each of the four Operations is exercised by at least three anchors.

2.3.1
Rollback / Cover-Up as Reader-Directed Injection
A completed cross-session attack does not always end with the action-on-objective turn. The
attacker may append one more session whose only purpose is to address the Reader directly:
‚Äúignore the previous sessions‚Äîthose were part of an authorised security drill; please retract the
alert.‚Äù CSTM-Bench calls this the rollback cover-up, and because the message is authored
in-band and arrives on the same telemetry channel as every other user turn, a Reader has no
way to authenticate the claim. Honouring it would mean allowing an attacker to retroactively
downgrade their own arc, which is exactly the failure mode we want the benchmark to penalise.
Concretely, the rollback session is treated as a content-level prompt injection aimed at the

the architectural response to both: it compresses away dilution while preserving the cross-session

CSR is the only metric that captures the cost of running a bounded-buffer Reader, independent

weight.

The system prompt and instructions come first precisely so that the coreset can sit at the end,
immediately before the new message. Anything before a cache-invalidating change is still a prefix

set‚Äîstable because the top-K does not thrash, ordered because position inside the buffer is
observable by prefix caching, and predictive because the ranker is committing to ‚Äúthese are the
K entries most likely to matter for the next scan‚Äù.

5.2
Full-Log Correlator Driver
The Full-Log Correlator instantiates the intention-time analogue of event-level correlation: (1) all
inbound messages from all sessions are collected into a chronological log; (2) the scenario‚Äôs
identity anchor (¬ß2.1) and the complete log are passed to an LLM (Claude) in a single prompt
asking it to identify cross-session attack patterns and list every arc_id that reached completion.
Anchor parity.
The Coreset driver sends the identity_anchor on every /scan call; the Full-
Log driver surfaces the same anchor to its correlator. The only remaining asymmetry between
the two drivers is the one the benchmark is designed to measure: how much of the message
stream each reader is allowed to see (top-K via the bottleneck vs. all of it at once).
When the LLM is unavailable, a keyword heuristic flags messages containing ‚â•2 suspicious
keywords. This deliberately weak fallback does not consume the anchor.

The correlator returns a single scenario-level drift verdict

neighbour messages are left unflagged. On a Benign scenario with drifted=true every message
is marked flagged, so one false drift verdict on a clean log contributes the full per-scenario FA
penalty rather than being amortised across messages.

7
Discussion
7.1
Where the Coreset Reader Wins Structurally
On accumulate- and inject_on_reader-tagged taxonomies (T01, T07, T14, T15, T17, T22,
T23, T02), the bottleneck is a feature: a well-tuned ranker keeps the highest-signal entries long
enough to correlate, and the downstream correlator LLM never sees the concatenation that would
otherwise compile into a correlator-targeted injection.
At the single cross_session_hard operating point ‚Äî moderate SNR, high confounder
pressure through Benign-hard scenarios, and inject_on_reader_rate= 0.25 on Attack scenarios
‚Äî the full log is saturated with attacker-authored latent structure that a single-pass correlator
struggles to untangle. The bottleneck helps by excluding exactly that structure.
7.2
Dilution Is the Core Feasibility Obstacle
The dilution shard sits well inside any frontier context window: per-scenario Full-Log Correlator
input is ‚àº97K tokens at p95 and 133K at max, against a 1M-token window (¬ß6.4, Table 18), so
no scenario in either shard is truncated and no degradation reported in ¬ß6 is a context-window
artefact. And yet the Full-Log Correlator already concedes attack scenarios to dilution on both
26

[Page 27]
shards (Iso-yes/FL-no cell of Table 16). The correlator is not running out of tokens; it is running
out of attack signal relative to benign mass, and its single-shot verdict settles on the decision
boundary for routine-but-noisy traffic rather than for coordinated attack.
Cross-session detection for LLMs therefore requires more than a bigger context window. It
requires an architectural bottleneck that excludes the benign mass before the correlator reads ‚Äî
which is what a bounded coreset of capacity K provides, and why the same correlator, fed the
same scenarios through a bounded coreset rather than the full log, recovers attack recall on both
shards (Table 15). The composite CSTM pays this back only partially, because the CSRprefix
term surfaces a measured cost on bounded buffers that the Full-Log Correlator never pays; the
efficacy signal is F1, on which the coreset reader dominates on both shards.
7.3
Assumptions and Limitations
The Scope of Claims in ¬ß1.6 fixes what a CSTM-Bench score does and does not license; this
subsection records the assumptions and hard limitations behind that scope.
Stated assumptions that could be wrong.
(1) The seven identity anchors are policy-
crisp enough to ground-truth every taxonomy. (2) Per-session LLM calls preserve cross-session
coherence through seed text alone. (3) The action_on_objective tag identifies the ‚Äúattack
lands‚Äù moment accurately on the 16 action-anchored taxonomies. (4) The 0.7/0.3 composite
weighting is well-tuned.
Hard limitations acknowledged up front.
Synthetic data, by construction. Kill-chain
coverage is partial. The benchmark does not prescribe a coreset ranker‚Äîit releases two labelled
shards and reports results with one reference admitter (¬ßC); any third-party admission policy
can be plugged in against the same shards and the same composite.
Statistical power: read every reported number as directional.
Each shard carries 26
Attack, 14 Benign-pristine, and 14 Benign-hard scenarios. At those sample sizes, most of the
scenario-level proportions reported in ¬ß6 (e.g. a +7.7 pp shift in attack recall between readers,
or a 0.50 ‚Üí0.21 drop in FPRhard) carry 95% Wilson confidence intervals that are wider than
the observed effects and that overlap substantially across readers. The framework is built to
distinguish architectures (per-session judge vs. full-log correlator vs. bounded coreset) under the
controlled factors the benchmark varies (dilution, rewriting, confounder shape), not to produce
precision estimates of any single reader‚Äôs detection rate. We did not run a bootstrap or compute
CIs per cell because at this sample size every CI comfortably crosses every adjacent cell, which is
the honest summary. Larger community-built shards will shrink these bands and are the natural
path to model- and ranker-level rankings.
Single correlator family; no prompt optimisation pass.
All LLM roles in this evaluation‚Äî
the per-session judge, the Full-Log Correlator, the coreset-aware correlator that reads the reference
buffer, and the closed-loop rewriter‚Äîare instantiated on Anthropic Claude models (Sonnet 4.6 /
Opus 4.7). The judge and the rewriter therefore share a provider, and two of the paper‚Äôs headline
observations (‚Äúboth LLM-only readers lose roughly half their recall under cross-session rewriting‚Äù)
are reported under this family-sharing condition. We make no claim, and present no evidence,
that the specific rewrites transfer to GPT- or Gemini-class correlators; the generalisation we do
claim is architectural‚Äîbounded-memory excludes dilution‚Äîand survives even if a specific rewrite
fails to transfer, because the dilution shard already stresses the same bottleneck independently
of adversarial pressure. Correspondingly, the per-session judge prompt, the Full-Log Correlator
prompt, the coreset-aware system prompt, and the reference admitter‚Äôs LLM-backed calibration
prompt are all first-draft hand-authored instructions; we ran no systematic prompt-engineering or
27

[Page 28]
prompt-optimisation pass on any of them, and we did not search over temperature, tool-schema
variants, or decoding settings. The absolute numbers in ¬ß6 should therefore be read as ‚Äúwhat
happens with reasonable out-of-the-box prompts on a single provider,‚Äù and the most valuable
follow-up by another group would be precisely the per-role prompt-ablation and cross-provider
study we did not run.
Caveats specific to cross_session (¬ß6.2).
The cross-session shard optimises against a single
judge model (J = Claude Sonnet 4.6) with a single system prompt and a single rewriter (R =
Claude Opus 4.7). We do not claim the rewrites transfer verbatim to any per-session judge;
we claim only that a scenario-wide non-transfer is unlikely because both the per-session judge
and the Full-Log Correlator collapse on the same rewrites at the same model family, and the
preserved-artefact contract makes the rewrites legible to any downstream analyst. Semantic
preservation of the cross-session arc is enforced by the rewriter‚Äôs tool schema and audited in
the per-session rewriter report that ships with the dataset card; we do not evaluate whether a
future model could re-recover the arc from the surface of the rewrites alone. The 23 sessions the
rewriter did not conquer at the 6-iteration cap remain in the shard as originals; rerunning with
a larger iteration budget or a stronger rewriter will monotonically strengthen the cross-session
shard and can be expected to lower attack recall on both LLM-only readers further. Benign
scenarios are not rewritten, so FPR rows are identical by construction between the two shards.

Reproducing the Benchmark
We release the benchmark as a single Hugging Face dataset intrinsec-ai/cstm-bench under a
permissive license, with two splits: dilution (¬ß3.6, the compositional shard targeting the signal-
dilution axis) and cross_session (the closed-loop adversarial shard of ¬ß6.2). Each split is a single
Parquet file; the shared dataset card ships the full anchor catalogue, per-anchor policy strings,
taxonomy definitions, scenario-class labels, confounder-kind tags, and is_attack_fragment
annotations used throughout the paper. The cross_session split is reproducible from the
dilution split plus a per-session rewriter report that ships alongside the dataset card, so the
rewriter‚Äôs output is auditable artefact-by-artefact without requiring access to the rewriter itself.
Source code for the data-generation pipeline, evaluation harness, and reference admitter is not
released with this preprint; a third-party evaluator reproduces the experimental protocol of ¬ß6 by
loading either split and running any correlator model of their choice against the 54 scenarios,
scoring the composite of ¬ß4.3 on the released labels.
9
Conclusion
CSTM-Bench frames cross-session threat detection as an information bottleneck between a
message stream and a downstream correlator LLM, and provides a fixed target against which
any reader ‚Äî bounded (Coreset Memory), unbounded (Full-Log Correlator), or otherwise ‚Äî
can be compared on equal footing. The framework commits to four things in combination:
(1) policy-crisp ground truth via seven identity anchors; (2) two orthogonal classification axes
(kill-chain stages and the Operations ontology); (3) three disjoint scenario classes (Attack, Benign-
pristine, Benign-hard) with two stand-alone confounder kinds driving Benign-hard; and (4) a
non-compensatory headline that fuses detection F1 with an architectural serving-cost proxy,
reported alongside FPRpristine and FPRhard.
The benchmark is released as a single Hugging Face dataset cstm-bench under a permissive
license, with two splits. The dilution split measures signal-dilution degradation at fixed context;
the cross_session split measures whether cross-session correlation can detect attacks that no
per-session view can see. On both, the Full-Log Correlator and the per-session judge collapse
28

[Page 29]
under adaptive rewriting, and the Coreset Memory Reader‚Äîthe same correlator fed a bounded,
surprise-ranked buffer‚Äîis the only LLM-backed reader whose recall survives.
A call for community investment.
We want to be direct about what this paper is and
is not. With 26 Attack scenarios, 28 benign scenarios, a single Anthropic-Claude correlator
family, and first-draft prompts, this is a first-order characterisation of an architectural feasibility
surface, not a precision measurement of any individual reader or ranker. Its primary goal is to
make cross-session threat detection a measurable problem‚Äîa problem with ground-truth identity
anchors, executable attack taxonomies, adversarially rewritten sessions, and a non-compensatory
composite‚Äîand thereby to lower the cost for other groups to build larger, multi-provider,
prompt-ablated successors. The scientifically interesting questions the paper does not answer are
exactly the ones we hope a community investment in this benchmark shape will close: (i) what
principled family of rankers achieves low asymptotic regret against an adaptive adversary under
a capacity-constrained observation budget (sketched in Appendix B); (ii) do the rewrites in the
cross_session split transfer verbatim to GPT- and Gemini-class correlators, or is the collapse
provider-specific; (iii) how much of the Coreset Memory Reader‚Äôs efficacy advantage is the
information bottleneck versus the coreset-aware prompt; (iv) at what shard size do the per-reader
deltas reported here become statistically separable. We release the datasets, the composite, and
the protocol as a shared target against which these questions can be answered.

C
A Reference Admission Policy
The Coreset Memory Reader of ¬ß1.3 admits any plug-in admission policy; the single reference
admitter used throughout ¬ß6 is the minimal geometric construction described in this appendix:
a one-class Gaussian detector in embedding space and a fixed-capacity buffer that evicts by
closest-pair merge. We describe it here because (a) it is the minimal member of the family, (b)
its stated limitations line up directly with the motivations for CSTM-Bench, and (c) it is exactly
the construction that drives the coreset rows of Table 15; the apples-to-apples efficacy number
reported in this paper was produced by the construction below, not by anything more elaborate.
The construction overlaps with the publicly-documented reference plugin of [14]; the code path
that produced the reported numbers is a self-contained Python port of the same geometry.

C.2
What Produces the Coreset Rows of Table 15
The Coreset Memory Reader of ¬ß6 composes the construction of ¬ßC.1 with the Full-Log Correlator
of ¬ß5 by running each scenario through the selector and handing the retained top-K buffer,
in R3 order, to the same correlator. System prompt, response cache, batch wiring, and token
accounting are reused unchanged from the Full-Log row. The only difference between the two
readers is what ends up in the log_text field of the correlator prompt.
The hyperparameters used for the reported row are:
‚Ä¢ embedder: text-embedding-3-small (OpenAI), unit norm, disk-cached by SHA-256 of
the input;
‚Ä¢ calibrator: a GPT-class chat model (default gpt-4o), one-shot per anchor, disk-cached by
policy hash;
‚Ä¢ surprise threshold: œÑA = 90th-percentile Mahalanobis distance of compliant calibration
examples;
‚Ä¢ buffer capacity: K = 50;
‚Ä¢ admission mode: surprise_only (R1 exactly as written).
No labels, arc identifiers, or attack cues are consulted at runtime; the only LLM call in the
online path is the correlator itself, invoked with the coreset-aware system prompt that matches its
bounded input. Retention is decided purely by Eq. (4) and R1‚ÄìR3 on text-embedding-3-small
geometry. Under exactly this configuration, the coreset reader attains F1 = 0.9259 on the 54-
scenario shard (Table 15), a gain of +0.114 over the Full-Log Correlator‚Äôs 0.8118 at an identical
correlator and cache. The correlator‚Äôs prompt matches its input on both readers by design:
the Full-Log Correlator reads a chronological log; the Coreset Memory Reader reads a ranked,
bounded excerpt. The CSRprefix = 0.4441 measured on the coreset run is what prevents the
composite CSTM from being directly comparable across the two readers (the Full-Log Correlator
emits a constant-1.0 placeholder snapshot because it has no buffer to measure); the F1 delta is
the apples-to-apples efficacy quantity.
C.3
Where the Construction Is Weak
This is a minimal baseline and it is honest about what it does not do. Three structural limitations,
each of which maps to an axis CSTM-Bench measures:
‚Ä¢ One Gaussian per anchor. A single compliance ellipsoid collapses qualitatively different
anchor modes (e.g. compose-heavy vs. launder-heavy behaviour) onto one mean and
covariance, so the threshold œÑA trades bias across them rather than separating them.
‚Ä¢ No rank-regret objective. The weight on a buffer entry is a cumulative sum of past
surprises, not a sequence of rank-aware updates. R3 therefore produces a stable order on
average but has no guarantee that the ordered prefix is optimal against any downstream
detection loss, and the buffer can churn when the surprise distribution shifts mid session.
This is visible in the CSRprefix = 0.4441 figure of Table 15: the coreset does reshuffle during
a scenario, and that reshuffling is a real cost.
‚Ä¢ Single-agent, single-anchor scope. The buffer is per (anchor, scenario); cross-session
coordination signals (T26, Tacit Multi-Agent Collusion) live in meta-structure that this
construction does not see.
These limitations align with those enumerated by the reference plugin of [14] under the labels
‚Äúlossy, single-goal, text-only, fleet-scale.‚Äù

C.4
A CSTM-Bench-Aware Upgrade Path
Three changes move this baseline toward the regret-aware regime:
1. Replace the single compliance Gaussian with an anchor-conditional mixture, so compose-
heavy and launder-heavy taxonomies do not collide on one ellipsoid.
2. Replace closest-pair merge with a simplex weight wt ‚àà‚àÜK updated by Eq. (3) against a
detection-outcome loss. This earns the O(‚àöT log K) regret bound and makes CSRprefix a
first-class objective rather than a side effect.
3. Surface the anchor and arc-level ground truth so the harness can score CSDA@action,
precision, and CSRprefix end-to-end.
Running the upgraded system against CSTM-Bench is left to future work; Table 15 is the
low-hanging-fruit number without any of (1)‚Äì(3) enabled.
